# 业务逻辑时序图

KMS的核心业务逻辑主要围绕**密钥管理**和**凭据管理**两大核心组件展开。以下分别展示数据加解密与凭据动态获取的核心业务时序图。

**密钥管理与数据加解密流程**

应用程序通过集成阿里云SDK或KMS实例SDK，调用KMS提供的密码运算API。KMS底层依托经权威认证的硬件安全模块（HSM）保障密钥的安全托管与运算，完成数据的加密、解密或数字签名后，将结果返回给应用，应用再将密文数据安全存储至各类云产品（如ECS云盘、OSS、RDS等）中。

```mermaid
sequenceDiagram
    participant App as 应用程序/用户
    participant SDK as 阿里云SDK/KMS实例SDK
    participant KMS as 密钥管理服务KMS
    participant HSM as 硬件安全模块(HSM)
    participant Cloud as 云产品/存储服务

    App->>SDK: 1. 发起数据加密/解密/签名请求
    SDK->>KMS: 2. 调用密码运算API (传入密钥ID和数据)
    KMS->>HSM: 3. 请求执行底层密码运算
    HSM-->>KMS: 4. 返回运算结果 (密文/明文/签名)
    KMS-->>SDK: 5. 返回密码运算结果
    SDK-->>App: 6. 返回处理后的数据
    App->>Cloud: 7. 存储密文数据或进行后续业务操作
```

**凭据管理与动态获取流程**

应用程序通过凭据管家客户端、凭据管家JDBC客户端或RAM凭据插件，向KMS请求托管的凭据（如RDS账密、RAM AK、ECS凭据等）。KMS解密并返回凭据，客户端在本地缓存后供应用使用。同时，KMS后台支持凭据的定期自动轮转，使应用程序规避明文配置风险，有效降低凭据泄露事件的危害。

```mermaid
sequenceDiagram
    participant App as 应用程序
    participant Client as 凭据管家客户端/JDBC客户端
    participant KMS as 密钥管理服务KMS (凭据管家)
    participant Target as 目标服务 (如RDS/ECS)

    App->>Client: 1. 请求获取目标服务凭据
    Client->>KMS: 2. 调用获取凭据API (传入凭据名称)
    KMS->>KMS: 3. 查找凭据并解密 (支持定期轮转机制)
    KMS-->>Client: 4. 返回明文凭据及版本信息
    Client->>Client: 5. 本地缓存凭据
    Client-->>App: 6. 返回凭据信息
    App->>Target: 7. 使用动态凭据建立连接并访问数据
    Note over KMS: 后台定期轮转凭据，<br/>更新版本并通知客户端刷新
```
