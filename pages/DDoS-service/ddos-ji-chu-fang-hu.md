# DDoS基础防护

## 产品对外文档：

### 服务介绍：

*   产品定位与演进历程
    
    *   DDoS基础防护是阿里云为部分核心云产品**免费集成**的网络层与传输层DDoS防御能力，定位于**普惠型、默认启用、零配置启动**的基础安全防线，不作为独立售卖产品，而是云产品原生能力的一部分。
        
    *   自2015年随ECS/SLB等首批云产品上线以来，持续演进：  
        ▪ 2017年支持EIP与IPv6网关；  
        ▪ 2019年引入AI智能基线分析，替代纯静态阈值触发机制；  
        ▪ 2021年扩展至WAF、GA、AnyCast EIP等新形态产品；  
        ▪ 2023年起支持按地域差异化黑洞阈值策略与动态防护能力弹性调整（基于历史攻击画像）。
        
    *   能力覆盖组件：直接内嵌于ECS、SLB、EIP（含NAT绑定EIP）、IPv6网关、轻量应用服务器、WAF、全球加速（GA）、AnyCast EIP等云产品实例中，**无独立部署组件**，由各云产品控制面统一纳管。

*   对外介绍架构图
    
    *   **无中心化独立服务架构**：DDoS基础防护无独立容器组或微服务，其清洗引擎、AI分析模块、阈值决策逻辑均深度集成于各云产品的转发面（如SLB的LVS集群、ECS的宿主机vRouter、EIP的边缘网关节点）及控制面（如SLB控制台API、ECS OpenAPI）。
        
    *   **数据流向**：互联网入向流量 → 云产品边缘接入点（如POP节点、EIP网关）→ 实时镜像至本地清洗模块 → AI基线比对 + 阈值判定 → 触发清洗（丢弃恶意包/重定向至清洗中心）或放行 → 正常流量转发至后端实例。
        
    *   **上下游依赖**：  
        ▪ 依赖Tianji（提供底层网络设备状态、黑洞下发通道）；  
        ▪ 依赖OpsApi（清洗阈值配置、黑洞事件上报、日志归集）；  
        ▪ 与云监控（CloudMonitor）联动实现攻击流量指标透出（如`ddos.attack.bps`）；  
        ▪ 不依赖VPC控制面（防护发生在VPC外侧边缘），但受VPC安全组规则最终生效影响（清洗后流量仍需过安全组）。
        
    *   参考：[[DDoS基础防护黑洞阈值|DDoS基础防护黑洞阈值]]、[[阿里云黑洞策略|阿里云黑洞策略]]

*   各核心组件能力详细说明
    
    *   **AI智能分析引擎**：运行于各云产品边缘节点，基于7天滚动业务流量基线（自动学习QPS、BPS、PPS、协议分布、源IP熵值等维度），实时计算偏离度，仅当AI判定为攻击 *且* 流量达用户设置阈值时才触发清洗，显著降低误清洗率。
        
    *   **动态清洗阈值模块**：支持用户在控制台手动设置BPS/PPS清洗触发阈值（上限受实例规格约束），阈值变更实时生效，无需重启实例；阈值存储于各云产品元数据库（如SLB的configdb、ECS的cmdb）。
        
    *   **黑洞管控模块**：当入向攻击流量超过该实例对应地域的黑洞阈值（如华东1为5 Gbps）时，通过Tianji下发黑洞指令至边缘网关，强制丢弃所有公网入向流量，持续时间由攻击强度动态决定（5–24小时），期间实例仍可出向访问。

*   与阿里云其他产品的关系
    
    *   与 VPC、ECS、SLB 等Top30产品的交互方式：  
        ▪ **VPC**：防护发生在VPC边界之外（即EIP/SLB公网IP侧），不干预VPC内网流量；黑洞期间VPC内网通信不受影响。  
        ▪ **ECS**：防护作用于ECS绑定的EIP或公网IP，清洗后流量经vRouter转发至ECS，ECS自身无需安装Agent；安全组规则在清洗后仍生效。  
        ▪ **SLB**：防护作用于SLB公网IP，清洗模块集成在SLB LVS集群中，清洗后健康检查流量仍可穿透，保障后端ECS可用性感知。  
        ▪ **影响边界**：DDoS基础防护异常（如AI误判、阈值失效）**仅可能导致单实例级误清洗或漏防**，不会引发跨VPC、跨账号、跨地域的级联故障；**不会导致VPC路由中断、SLB会话保持失效、ECS内网IP冲突等非DDoS领域问题**。

### QA（高频问答）：

*   Q：DDoS基础防护是否需要单独购买或开通？  
    A：否。只要使用支持的云产品（如ECS、SLB、EIP），且实例处于支持地域，即**默认开启、不可关闭、完全免费**。

*   Q：能否自定义防护策略（如只防SYN Flood，不限制UDP）？  
    A：否。基础防护为全量协议清洗，不支持协议级策略定制；如需精细化策略，请升级至[[DDoS原生防护|DDoS原生防护]]。

*   Q：AI分析是否会导致隐私泄露？  
    A：否。AI仅分析脱敏后的流量统计特征（如每秒包数、源IP分布熵值），**不采集、不存储原始报文内容、不记录用户业务数据**。

*   Q：黑洞期间，我能否通过内网或跳板机访问该ECS？  
    A：可以。黑洞仅屏蔽**公网入向流量**，VPC内网、经典网络内网、跳板机SSH等内网访问路径完全不受影响。

*   Q：清洗阈值设置后，是否立即生效？  
    A：是。阈值配置通过云产品OpenAPI下发，通常在**10秒内全网生效**，无需重启实例或服务。

### 组件接口人、研发负责人等角色信息：

*   **统一接口人**：DDoS基础防护无独立研发团队，由各云产品线安全模块共担责任：  
    ▪ ECS侧：ECS网络安全部（L2运维权限覆盖阈值配置、黑洞状态查询）  
    ▪ SLB侧：SLB转发面组（L2可执行`slb describe-ddos-status`，L3支持清洗日志深挖）  
    ▪ EIP/IPv6网关侧：云网络-边缘网关组（L3负责黑洞策略调优与AI模型迭代）  
    ▪ **紧急升级路径**：值班发现AI持续误判或黑洞异常频发 → 提交Aone工单至「Anti-DDoS/基础防护」标签 → 自动路由至对应云产品L3接口人（SLB/ECS/EIP三选一，依据攻击入口判断）

## 方案支撑文档：

### 运维指导/运维手册

*   常用的数据库，常用表（写明存储什么信息）  
    ▪ SLB：`slb_configdb`.`ddos_protection_policy`（存储实例级清洗阈值、开关状态）  
    ▪ ECS：`ecs_cmdb`.`instance_ddos_config`（存储EIP绑定关系与黑洞状态）  
    ▪ EIP：`eip_meta`.`blackhole_history`（黑洞触发时间、持续时长、峰值BPS）

*   关键日志路径，组件，内容，轮转策略  
    ▪ `/var/log/aliyun/ddos/basic/`（所有支持产品的统一日志根目录）  
    ▪ `cleaning.log`：清洗触发详情（时间、源IP段、攻击类型、清洗丢弃包数）→ 按日轮转，保留7天  
    ▪ `ai_baseline.log`：AI基线更新记录（每日02:00生成新基线，含QPS/BPS标准差）→ 按周轮转，保留4周  
    ▪ 集群范围：**仅存在于部署了EIP/SLB/ECS公网实例的物理机或边缘节点**，非全量集群采集。

*   问题排查SOP，通用场景的排查思路和路径  
    ▪ **现象：业务间歇性不可达** → 查云监控`ddos.attack.bps`是否突增 → 查`ddos.basic.cleaning.triggered`告警 → 登录对应云产品控制台查看清洗状态 → 检查`cleaning.log`确认攻击源IP段  
    ▪ **现象：突然无法公网访问** → 查`ddos.basic.blackhole.active`告警 → 查`eip_meta.blackhole_history`确认黑洞起止时间 → 使用内网验证是否仅公网中断  
    ▪ **现象：阈值设置不生效** → 查`ddos.basic.threshold.exceeded`告警 → 查`slb_configdb.ddos_protection_policy`确认写入值 → 核对实例规格文档确认阈值上限  

*   版本升级指南：DDoS基础防护能力随云产品版本自动升级，无需人工干预；重大AI模型更新通过灰度发布，周期约季度一次，公告同步至[[阿里云公告中心|阿里云公告中心]]。

### 典型问题排查解决方案

```yaml
（针对“清洗触发后业务延迟升高”）
一、问题描述
● 问题现象：SLB监听端口响应延迟从50ms升至2s+，监控显示`ddos.basic.cleaning.triggered`告警，`cleaning.log`中大量SYN包丢弃记录。
● 适用范围：所有SLB实例（2022年及以后版本），地域全覆盖，影响范围为该SLB下全部后端ECS。

二、排查信息收集
● 必须收集的信息：SLB实例ID、触发时间点（精确到秒）、后端ECS内网IP列表、`cleaning.log`对应时间段完整日志。
● 检查终态的方法：登录SLB控制台 → 实例详情页 → “DDoS防护”页签 → 查看“当前清洗状态”及“最近清洗详情”。
● 排查问题步骤：
  - 步骤1：在`cleaning.log`中搜索`"attack_type":"SYN_Flood"`，确认是否为真实攻击；
  - 步骤2：对比同一时段`slb_monitor`.`qps`指标，若QPS无突增，则大概率为误清洗；
  - 步骤3：检查`ai_baseline.log`中该时段`baseline_bps`与`actual_bps`偏差是否>300%，确认AI基线是否失准。

三、解决步骤
 场景一：确认为真实SYN Flood攻击
 - 适用条件：`cleaning.log`中攻击源IP高度分散（>1000个C段）、`slb_monitor.qps`平稳、`actual_bps`远超`baseline_bps`
 - 实施步骤：
   - 登录SLB控制台 → DDoS防护页签 → 将BPS阈值临时上调至规格上限（如1 Gbps实例设为950 Mbps）
   - 执行命令：`aliyun slb SetDDoSStatus --LoadBalancerId <id> --BpsThreshold 950000000`
 - 结果验证：`cleaning.log`停止新增SYN丢弃记录，延迟恢复至50ms内。

 场景二：AI基线失准导致误清洗
 - 适用条件：`cleaning.log`攻击源IP集中（<10个IP）、`slb_monitor.qps`同步飙升、`actual_bps`仅略超`baseline_bps`
 - 实施步骤：
   - 登录SLB控制台 → DDoS防护页签 → 点击“重置AI基线”
   - 执行命令：`aliyun slb ResetAIBaseline --LoadBalancerId <id>`
 - 结果验证：`ai_baseline.log`生成新基线，`cleaning.log`不再触发。

四、非本产品排查
● 若`cleaning.log`显示`"attack_type":"HTTP_Flood"`：DDoS基础防护不防应用层，此为WAF或DDoS原生防护职责，需转交[[WAF产品组|WAF产品组]]。

五、快速定位工具
● 脚本位置：`/opt/aliyun/ddos-tools/clean-diagnose.sh`
● 使用方法：`bash /opt/aliyun/ddos-tools/clean-diagnose.sh --slb-id lb-xxx --start-time "2024-05-20 14:00:00"`
```

### 紧急场景止血与恢复手册：

*   **黑洞期间应急**：  
    ▪ 止血：立即通过内网或跳板机登录受影响ECS/SLB，确认业务核心链路是否正常；  
    ▪ 恢复：黑洞为自动解除机制，**不可人工提前退出**；可通过`aliyun eip DescribeEipMonitorData`轮询`BlackHoleStatus`字段，状态变更为`inactive`即恢复；  
    ▪ 替代方案：若业务强依赖公网，紧急将EIP解绑并绑定至已开启[[DDoS原生防护|DDoS原生防护]]的备用EIP（需提前准备）。

*   **AI持续误判应急**：  
    ▪ 执行`aliyun slb ResetAIBaseline`重置基线（SLB）或`aliyun ecs ResetAIBaseline`（ECS）；  
    ▪ 若重置后2小时内再次误判，立即提交Aone工单至「Anti-DDoS/基础防护」并标注【P0-AI误判】，触发L3模型热修复。

### 横向研发文档：

*   接入指引：云产品接入DDoS基础防护需满足——  
    ▪ 在边缘网关节点部署`ddos-basic-engine`轻量模块（提供BPF过滤+AI推理SDK）；  
    ▪ 控制面OpenAPI需支持`SetDDoSStatus`/`DescribeDDoSStatus`接口；  
    ▪ 元数据表需包含`ddos_config`字段存储阈值与开关状态。  
    ▪ 详细规范见[[DDoS基础防护接入规范|DDoS基础防护接入规范]]

*   产品对接方案细节：  
    ▪ 清洗决策必须在100ms内完成（含AI推理），超时则降级为静态阈值模式；  
    ▪ 所有清洗日志必须打标`product=<slb/ec2/eip>`并推送至SLS `aliyun-ddos-basic`日志库；  
    ▪ 黑洞事件必须通过Tianji标准事件总线广播，事件格式遵循`{event: "blackhole_active", resource_id: "eip-xxx", duration: 3600}`。

*   产品对接范围：当前仅支持**具备公网IP属性**的云产品，不支持RDS、Redis等纯内网产品；未来扩展需满足“流量必经边缘网关”前提。

## 产品对内文档：

### 完整架构图：

*   系统层级：  
    ▪ 边缘层：各云产品POP节点部署`ddos-basic-engine`（含BPF过滤器 + 轻量AI推理引擎）；  
    ▪ 控制层：SLB/ECS/EIP控制台调用各自OpenAPI，配置写入元数据库；  
    ▪ 分析层：集团AI平台每日拉取脱敏流量特征，训练基线模型，下发至边缘节点；  
    ▪ 决策层：边缘节点实时运行模型，输出`anomaly_score`，与阈值比对后触发清洗或黑洞。

*   已知问题：  
    ▪ 多EIP绑定同一ECS时，AI基线会混淆不同EIP流量，导致误判（已列入2024 Q3优化计划）；  
    ▪ IPv6网关在双栈模式下，AI仅学习IPv4基线，IPv6流量清洗准确率下降30%（临时方案：IPv6阈值设为IPv4的1.5倍）。

### 业务逻辑时序图

*   用户使用：  
    用户创建SLB → 控制台默认开启DDoS防护 → 边缘节点自动初始化AI基线（7天学习期） → 攻击流量到达 → BPF镜像流量至引擎 → AI计算`anomaly_score` → `score>0.8 && bps>threshold` → 触发清洗 → 日志上报OpsApi → 云监控告警。

*   工作流流转：
