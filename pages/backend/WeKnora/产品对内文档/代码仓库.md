# 代码仓库

*   **基线仓库**：主干代码仓 [[https://github.com/Tencent/WeKnora|WeKnora GitHub 仓库]]，采用 `main` 分支作为稳定基线，所有功能开发基于 `feature/*` 分支，经 CI 验证后通过 PR 合并至 `main`；版本发布遵循语义化版本（SemVer）规范，Tag 标记格式为 `vX.Y.Z`，与 Helm Chart、Docker 镜像、CLI 二进制等制品版本严格对齐。

*   **代码结构**：项目采用单体多模块、关注点分层的架构设计，核心语言为 Go（占比约 68% 文件数），辅以 TypeScript（前端）、Python（文档解析/评估脚本）、SQL（Schema 定义、迁移脚本与查询模板）、YAML（CI/CD 流水线与配置）。模块组织严格遵循领域驱动设计（DDD）原则与清晰职责边界：
    - `cli/`：统一命令行客户端，支持 `agent`、`kb`、`search`、`mcp`、`api` 等子命令，实现本地知识操作、MCP 工具调用与轻量交互；
    - `internal/`：核心服务层，按 DDD 分层组织：
        - `application/service`：业务用例编排与协调；
        - `domain/models`：领域模型定义（含知识图谱实体、关系、文档元数据等）；
        - `infrastructure/`：基础设施适配层，涵盖 `retriever/`（向量/图/关系混合检索器适配器，支持 Qdrant、Neo4j、Doris、Milvus、Elasticsearch、Tencent VectorDB、Weaviate、PostgreSQL、SQLite）、`im/`（飞书/企微/Slack/Telegram 等 IM 协议适配器）、`docparser/`（文档解析器集成层，引用 `docreader/proto/` 契约）、`models/`（LLM/Embedding/Rerank/VLM/ASR 多模型抽象与插件化实现）；
        - `handler/` 与 `router/`：HTTP/gRPC 接口层，负责请求路由、参数绑定与响应封装；
    - `docreader/`：独立、可复用的文档解析子系统，支持 PDF/Word/Markdown/PPTX/Excel 等主流格式解析、语义分块（`splitter/`）、结构化建模（`models/`）、元数据提取，并通过 `proto/`（gRPC 接口定义）对外提供标准化契约；其能力被 `internal/infrastructure/docparser/` 明确依赖，形成清晰上下游解耦；
    - `frontend/`：TypeScript + Vue/Wails 构建的桌面/Web 混合前端，基于 Wails 实现嵌入式 Web UI，通过 `wailsjs/go/main/` 与 Go 后端深度集成；
    - `docker/`、`helm/`：标准化部署资产；`docker/` 提供 Docker Compose 快速启动方案；`helm/` 提供企业级 Kubernetes 生产部署的 Helm Chart，支持多租户、资源隔离、可观测性（Prometheus/OpenTelemetry）与 RBAC 配置。

*   **制品仓库**：构建产物统一托管于多平台，确保可追溯与一致性：
    - **Docker 镜像**：私有托管于 [腾讯云容器镜像服务 TCR](https://console.cloud.tencent.com/tcr)，公开镜像发布至 [Docker Hub `tencent/weknora`](https://hub.docker.com/r/tencent/weknora)，同时亦同步至 GitHub Container Registry（`ghcr.io/tencent/weknora-server`、`ghcr.io/tencent/weknora-cli`）；所有镜像标签均与 Git Tag（`vX.Y.Z`）严格对齐；
    - **Helm Chart**：发布至 `https://tencent.github.io/weknora/helm-charts`（GitHub Pages）；
    - **CLI 二进制**：Linux/macOS/Windows 平台可执行文件通过 GitHub Releases 分发，并由 `Formula/` 提供 Homebrew 支持；
    - **其他制品**：Go 编译二进制（`weknora-server`, `weknora-cli`）、前端静态资源包（`weknora-frontend`）、SQL 迁移脚本与 DuckDB 内置数据集（见 `cmd/download/duckdb/`）均由 GitHub Packages 托管。

*   **关联依赖仓库等**：
    - **上游模型与存储服务**：依赖 [[https://github.com/ollama/ollama|Ollama]]（本地 LLM）、[[https://github.com/qdrant/qdrant|Qdrant]]（向量库）、[[https://github.com/neo4j/neo4j|Neo4j]]（图谱存储）、[[https://github.com/apache/doris|Doris]]（关系型混合检索）、[[https://github.com/milvus-io/milvus|Milvus]]、[[https://github.com/elastic/elasticsearch|Elasticsearch]]、[[https://github.com/weaviate/weaviate|Weaviate]]、[[https://www.postgresql.org/|PostgreSQL]]、[[https://www.sqlite.org/|SQLite]] 等，其接入逻辑统一抽象于 `internal/infrastructure/retriever/`；
    - **协议标准**：深度遵循 [[https://github.com/modelcontextprotocol/specification|MCP 规范主仓库]]，[[backend/WeKnora/index|WeKnora]] 实现 `mcp-server-go`，相关协议定义、服务端与客户端实现在 `internal/mcp/` 与 `cli/internal/mcp/`（及 `cli/cmd/mcp/` 工具链）；
    - **第三方 connector**：飞书、Notion、语雀等多源数据连接器位于 `internal/datasource/connector/`，各目录（如 `feishu/`、`notion/`、`yuque/`）均独立维护其 OAuth2 认证、增量同步、变更监听与错误重试逻辑，不直接依赖外部 SDK 仓库，仅对接其 OpenAPI 文档与认证流程；
    - **文档解析生态**：`docreader/` 子系统已沉淀为独立可复用组件，其 `proto/` 接口定义构成稳定契约，被 `internal/infrastructure/docparser/` 显式引用，支撑解析能力的模块化演进与跨服务复用。
