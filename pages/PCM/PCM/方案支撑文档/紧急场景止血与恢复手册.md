# 紧急场景止血与恢复手册

当遇到业务访问报错，且怀疑是 PCM 禁用 AK（AccessKey）导致时，请严格按照以下流程进行应急止血与恢复操作：

1. **日志排查与判定**：优先通过各类网关的拦截日志判定报错是否由 AK 被禁用引起。
2. **提取请求 AK**：从拦截日志中准确提取出发起请求的 AK（AccessKey ID）。
3. **状态确认**：通过 PCM 服务查询该 AK 的当前状态，确认其是否已被禁用。
4. **应急止血处置**：若确认 AK 已被禁用且影响核心业务，立即采用应急处置方案进行恢复（如重新启用该 AK 或切换至备用 AK），以快速恢复业务访问。
5. **根因排查与反馈**：止血完成后，及时将问题反馈至研发侧，排查 AK 被禁用的具体原因，避免问题复发。

## 各网关拦截日志特征与 AK 提取指南

在排查过程中，可参考以下常见网关的拦截日志特征来快速定位问题并提取 AK：

| 网关类型 | 拦截状态码 | 核心错误特征 (Error Code / Message) | AK 提取字段 |
| --- | --- | --- | --- |
| **OSS** | `403` | `"error_code": "InvalidAccessKeyId"` | `access_id` |
| **SLS_INNER** | `401` | 无特定 Error Code，主要依赖状态码判定 | `AccessKeyId` |
| **SLSPUB** | `401` | `"ErrorCode": "Unauthorized"`<br>`"ErrorMsg": "AccessKeyId is disabled: <AK>"` | `AccessKeyId` |
| **ASAPI** | `500` / `failed` | `"errorCode": "asapi.server.request.parameter.accesskeyid.error"`<br>`"errorMessage"` 包含 `"The Access Key is disabled."` | `accessKeyId` |
| **OSS_OCM_INNER / KMS** | `401` / `403` | 关注 `InvalidAccessKeyId` 或 `AccessKey disabled` 等关键字 | 参考对应网关标准字段 |
