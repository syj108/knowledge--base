# QA（高频问答）

**Q：什么是 PCM（[[PCM/平台凭证管理服务/index|平台凭证管理服务]]）？它的核心目标是什么？**
**A：** PCM（Platform Credential Management）是 `baseServiceAll` 中的基础服务。其核心目标是**接管平台底表 AK，实现凭证的动态轮换与安全管控**，从而提升平台的整体安全性。

**Q：PCM 中的底表 AK、initAK、IAMID、secretARN 和 GatewayCode 分别指什么？**
**A：** 
- **底表 AK**：通过全局变量方式声明、云平台初始化时自动创建的 AK。
- **initAK**：原始底表 AK，即 PCM 改造前应用直接使用的凭证。
- **IAMID**：产品申请派生时的身份标识。格式为 `${CLUSTERNAME}:<serverrole名称>`，PaaS 格式为 `{{ .Values.productName }}:{{ .Release.Name }}`（当前未强校验格式）。
- **secretARN**：凭证目标资源标识，格式为 `apsara:pcm:akid:<accessKeyId>:dst_endpoint:<GatewayCode>:sk:<accessKeySecret>`。
- **GatewayCode**：服务的认证网关 code，用于区分 AK 私用网关和标准 AK 认证网关（当前版本仅标准 AK 认证网关支持使用底表 AK）。

**Q：标准 AK 认证与 AK 私用场景有什么区别？**
**A：** 
- **标准 AK 认证**：AK 生命周期在 UMM 中保管，标准网关通过对接 UMM 进行 AK 签名校验（如 POP、OpenAPI、OSS）。当前访问标准 AK 认证服务的云产品均已适配完成。
- **AK 私用场景**：服务不接或无法接 UMM，直接把 AK 参数记录到本地配置文件或数据库中，请求过来时用本地配置校验。当前尚未强制要求适配，已适配的产品通过 PCM 服务兑换出原始底表 AK。

## 派生 AK 队列机制

**Q：派生 AK 队列机制是如何运作的？**
**A：** 底表在生成派生 AK 时，每个派生 AK 会关联一个派生 AK 队列。队列默认维持 **7 把有效派生 AK**，每把派生 AK 有效期为 **24 小时**。因此，一把派生 AK 从创建到默认过期需要 7 天。队列会持续轮转（定期创建新 AK、禁用老 AK）。

**Q：派生 AK 队列有哪些划分级别？为什么不推荐使用 ClusterName 级别？**
**A：** 派生 AK 队列有两种划分级别：
- **initAK 级别（默认且推荐）**：一个底表 AK 对应一个派生 AK 队列，全局共享。
- **ClusterName 级别（不推荐）**：按集群划分，同一集群内一个底表 AK 对应一个派生 AK 队列。

**不推荐 ClusterName 级别的原因**：在 UMM AK 管理中，每个账户对应的 AK 数量有上限。按 ClusterName 级别划分时，每个集群都会为同一个底表 AK 创建独立的派生 AK 队列，多集群叠加极易把账户的 AK 上限打满，导致无法创建新的派生 AK。

**Q：派生 AK 队列在哪些情况下会触发保护机制并暂停轮转（或显示“轮转状态已停止”）？**
**A：** 为保护正在使用中的凭证，队列在以下情况下会暂停轮转：
1. **产品最新派生 AK 保护（有产品仍在使用第 7 把 AK）**：当要禁用队列里最早的 AK 时，若该 AK 是某个产品获取的“最新”派生 AK（即该产品拿到后未再获取新 AK），队列将停止轮转，直到其他产品获取了更新的派生 AK。
2. **产品未及时更新**：使用该产品的队列中，有产品未及时更新获取新 AK（参考 平台凭证管理服务（PCM）介绍）。
3. **平台 AK 访问日志不可行保护**：当访问日志不可行时，PCM 无法确认即将禁用的派生 AK 是否仍有产品在调用，将在第一把队列即将禁用时停止轮转。
4. **平台 AK 访问日志保护（日志可信时）**：在准备禁用某把派生 AK 前，系统会检查平台 AK 访问日志。若日志显示该 AK 当前仍在被使用，则停止轮转。
5. **手动关闭自动轮转**：IAMID 中包含 `CLOSE_AUTO_ROTATE` 状态，表示该队列默认不轮转。

**Q：什么场景下需要手动创建派生 AK（临时 AK）？**
**A：** 当某个应用需要使用临时 AK 登录，或者使用的 initAK 被禁用时，可以手动创建临时 AK 使用。

**Q：创建临时 AK 时，申请者 ID（IAMID）有什么格式要求？如果提示已存在怎么办？**
**A：** 申请者 ID（IAMID）是服务的身份标识，常规格式为“集群 + `:` + sr”拼接而成（例如：`StandardCloudCluster-A-20250906-00bf:PcmController`）。如果系统提示该 ID 已存在，可以在其后拼接任意字符串以作区分。

**Q：创建临时 AK 成功后，如何获取 SK 明文？如果不小心关闭了弹窗怎么办？**
**A：** 临时 AK 对应的 SK 明文仅在创建成功后的弹窗内展示，关闭弹窗后系统内不再显示。因此创建成功后请立即复制保存。如果不慎关闭弹窗，由于系统不对外提供 SK 明文信息查询能力，您需要重新创建临时 AK。

**Q：AK 申请日志的记录频率是怎样的？**
**A：** AK 申请日志记录了每个 IAMID 申请派生 AK 的记录。由于 pcm-core 中针对每个 IAMID 的底表 secretARN 的缓存时间为 12 小时，对于一直在使用派生 AK 的产品，理论上每 12 小时会有一条申请记录。

**Q：在 AK 申请详情中，显示“认证状态失败”会影响申请结果吗？**
**A：** 不会影响。“认证状态失败”仅表示 IAMID 不规范，不会对 AK 的申请结果产生任何影响。

## 管控模式与兼容策略

**Q：PCM 支持哪几种管控模式？**
**A：** PCM 支持以下四种管控模式：

| 模式 | 含义 | 行为 | 适用场景 | 版本 |
| --- | --- | --- | --- | --- |
| **None（默认）** | 不受 PCM 管理 | AK 正常使用，PCM 不介入 | 尚未改造的存量凭证 | / |
| **CompatibilityMode（兼容模式）** | 部分完成改造 | 提供轮换能力，但不对旧 AK 禁用 | 改造中的过渡态 | v3182-2510 |
| **StrictMode（严格模式）** | 使用方改造完成 | 新部署严格托管；热升级/扩等场景自动降级为兼容模式 | 存量改造完成后的目标终态 | v3182-2515以后 |
| **initStrictMode（初始严格模式）** | 新建凭证即完成改造 | 任何场景都开启严格处理 | 新增收口凭证 | v320 |

**Q：在热升级场景下，PCM 的兼容策略是什么？**
**A：** 
- **新部署项目**：根据 `restrict` 取值禁用原始通用能力，应用使用凭证进入定时轮换状态。
- **热升级项目**：原始凭证**不禁用**其通用能力，进入定时轮换状态；如需禁用老凭证，需通过观测日志在运维控制台灰度进行。
- **非 PCM 托管凭证**：一切照旧；若使用了 PCM SDK/CLI 但未被托管，将入参 initAK 返回让应用继续使用。

**Q：底表 AK 被禁用后，对业务可用性有什么潜在风险？**
**A：** 底表 AK 被 PCM 禁用后，产品的凭据供给完全依赖 PCM 链路（Core + Controller）。对于本地有缓存的运行中服务暂时无影响，但重启的服务如果此时 PCM 不可用，将拿不到任何有效凭据（底表已禁、派生获取失败、本地无缓存），会导致业务直接中断。因此，在严格模式下需确保 PCM 链路的高可用。

**Q：如何在控制台管理底表 AK？能否直接禁用底表 AK？**
**A：** 在 PCM 控制台中，可以查询底表 AK 的禁用状态以及启用底表 AK。但控制台未提供白屏禁用底表 AK 的能力，如需禁用底表 AK，请参考相关的变更文档进行操作。

## 高可用、容错降级与性能影响

**Q：如果 PCM Core 宕机或出现异常，SDK 会如何降级处理？对业务有何影响？**
**A：** PCM 设计了完善的高可用与容错逻辑，具体降级行为及业务影响如下：

| 场景 | SDK 行为 | 业务影响 |
| --- | --- | --- |
| 新部署时 PCM Core 还未 ready | 将入参作为返回 | 无影响（Core 未禁用老 AK） |
| 运行时 PCM Core 挂了 | 返回上次获取的老凭证（未在窗口期末尾） | 无影响 |
| 产品独立升级，PCM 未 ready | 将入参作为返回 | 无影响 |
| PCM 和应用都挂了需重拉（SDK 缓存未丢失） | 返回上次获取的老凭证 | 无影响 |
| PCM 和应用都挂了需重拉（SDK 缓存丢失） | **需先恢复 PCM 或使用老凭证应急脚本** | **业务中断** |

**Q：接入 PCM 后出现大量报错日志，是否有影响？**
**A：** 在 2507 版本 PCM 服务端尚未部署时，部分适配了 PCM 的产品可能访问 PCM 报错，但因降级返回了原始底表 AK，**不影响业务调用**。如果调用非常频繁，可能产生大量错误日志。此外，部分产品升级至 3186-2510 及以上版本，但 `baseServiceAll` 未升级，也可能出现同样问题。这类 WARN 日志不影响业务（SDK 已降级返回原始凭证），主要影响是可能触发客户端告警监控。

**Q：接入 PCM 后对时间敏感服务有什么影响？如何处理？**
**A：** 接入 PCM 后可能导致部分时间敏感服务延迟加大，且网络可能出现延迟。对于时间敏感服务，SDK 增加了超时策略。在 1.13-SNAPSHOT (20250908) 及以上版本中，支持通过 `PCM_TASK_DELAY` 环境变量设置访问 PCM 的最大超时时间（单位 ms，默认 1000ms 即 1s）。

**Q：PCM 各核心组件（SDK、Core、Controller）具备哪些关键安全与高可用特性？**
**A：**
- **PCM SDK / CLI（凭证获取端）**：支持**多级缓存**（本地内存、磁盘均有缓存）；具备**容错降级**能力，初始化异常时返回入参凭证，或返回最近一次获取的缓存凭证。
- **PCM Core（缓存中间网关）**：支持**本地缓存与定时同步**，减少直接访问 Controller 的频率；具备**缓存隔离**与**降级保护**，Core 宕机时 SDK 仍可返回上次获取的老凭证；作为中间层有效**缓解 Controller 压力**。
- **PCM Controller（策略中心）**：负责**凭证队列管理**与**模式管控**；支持**松→紧变更不自动生效**（需人工处理防误操作）；支持**灰度禁用**老凭证；提供**白屏管控（PKM）**、**日志查询关联**及凭证**状态管理**能力。

## 运维与辅助工具

**Q：如何判断底表 AK 或派生 AK 是否被禁用？**
**A：** 
- **底表 AK**：可以直接通过 PCM 控制台查询，或参考 PCM 运维手册进行查询。
- **派生 AK**：当前输出版本（3186、320）默认均不禁用派生 AK。如需查询，控制台仅可查询每个队列最近 14 把派生 AK，或通过数据库（clm_db 实例的 pcm_db 库中的 `ak_info` 表）进行查询。

### 网关日志查询工具

**Q：网关日志查询工具的 SLS 访问凭证如何配置？**
**A：** 配置中未自动适配 PCM 轮转，需要直接填写 PCM 轮转后的 AK。请通过 PCM 控制台手动获取派生 AK，并填入配置文件的 `access_key_id` 和 `access_key_secret` 中。

**Q：网关日志查询工具支持哪些运行模式？**
**A：** 支持两种运行模式：
- `scan`：全量扫描网关中底表 AK 的调用记录。
- `query`：关键字查询模式，支持通过网关代码和关键字（如事件 ID）查询日志详细信息。

**Q：使用 `scan` 模式扫描出的 AK 调用记录保存在哪里？**
**A：** 扫描记录默认自动存储在相对路径的 `output/scan_result_{时间戳}.csv` 文件中。输出路径和格式（支持 print、json、csv、all）可以在配置文件的 `output` 节点中进行自定义修改。

### 底表 AK 黑屏操作工具

**Q：运行底表 AK 黑屏操作工具时提示“错误: 缺少环境变量”怎么办？**
**A：** 该工具依赖环境变量来获取 PCM Controller 域名和生成请求签名。运行前请确保已在系统环境中正确配置 `pcm_ctrl_domain` 和 `pcm_rs` 这两个环境变量。

**Q：底表 AK 黑屏操作工具支持哪些具体操作？**
**A：** 支持以下 5 种操作指令：
- `enable`：启用指定的 AK（必须配合 `--ak` 参数使用）。
- `disable`：禁用指定的 AK（必须配合 `--ak` 参数使用）。
- `enable-all`：获取并启用全部底表 AK。
- `disable-all`：获取并禁用全部底表 AK。
- `query`：通过账号 ID 查询对应的 AK（必须配合 `--account-id` 参数使用）。

## 常见问题与故障排查

**Q：AK 调用网关被拦截（报 AK 被禁用/无效/不存在）如何排查和恢复？**
**A：** 首先从网关日志取出被拦截的 AK ID，在控制台或数据库查询是底表 AK 还是派生 AK。
- **底表 AK 被拦截**：说明 SDK 未成功获取派生 AK 走了降级逻辑，或未适配底表 AK。**恢复方法**：在 PCM 控制台启用该底表 AK 恢复业务，然后查 SDK 日志 code 确认降级场景。
- **派生 AK 被拦截**：说明派生 AK 已被轮转禁用，产品未及时更新。**恢复方法**：通常重启服务刷新 AK 即可，或停止该队列轮转；若无法重启，需手动启用 AK（参见 PCM应急处置）。最可能原因是半轮转模式下仅获取一次未持续轮转。

**Q：SDK 报错 signature error (HTTP 403) 如何排查？**
**A：** 签名验证失败通常由以下原因导致：
- **单元 region 报错，中心正常**：99% 是 regionid 传错，导致两端 IKM 计算不同。
- **中心和单元同时报错**：initAK/initSK 值本身错误。
- **个别产品报错**：`pcm_endpoint` 配置错误（域名拼写错误或多了路径）。
- **都确认正确仍 403**：环境中 SK 是加密存储的，产品未解密就传给了 SDK，导致签名 key 两端不一致。需确认产品侧调用 SDK 前已解密 SK。

**Q：Core 返回 HTTP 502 或触发限流如何排查？**
**A：** 大概率是触发了限流。PCM Core 的限流策略基于客户端 IP，同一台机器上运行多个产品组件时，高频产品可能耗尽 IP 限流配额导致同 IP 下其他产品被误伤。
排查步骤：
1. 检查 access.log 中 `limit_req_status` 字段。
2. 使用 `tsar -l -i 1 --nginx` 查看 QPS。
3. 调整限流配置：`/services/platform-credential-management/user/pcm_conf/pcm_core.json`。
4. 阈值参考（单核）：x86=200r/s, aarch64=189r/s, sw64=80r/s。

**Q：排查 PCM Core 日志时有什么注意事项？如何排查 error 和 access 日志？**
**A：** 
- **注意事项**：PCM 部署在两个 docker 上，日志排查时必须同时查询这两个 docker 的日志。
- **排查 error 日志**：
  - 有具体 requestid：
    ```bash
    grep -rn "您的requestid" /opt/tengine/logs/error.日期.log
    ```
  - 无具体 requestid（根据 akid、iamid 和时间段复合筛选）：
    ```bash
    grep "akid" /opt/tengine/logs/error.日期.log | grep "iamid" | awk '$1 >= "日期" && $2 >= "开始时间" && $2 <= "结束时间"'
    ```
- **排查 access 日志**：
  - 有具体 requestid：
    ```bash
    grep -rn "您的requestid" /opt/tengine/logs/access.日期.log
    ```
  - 无具体 requestid（根据 akid 和时间段复合筛选）：
    ```bash
    grep -E '"time_local": "(日期:小时:分钟|日期:小时:分钟)' /opt/tengine/logs/access.日期.log | grep "akid"
    ```

**Q：CLI 工具报错 ResponseParseFailure 如何处理？**
**A：** 原因是 `pcm_endpoint` 地址不对，响应 200 但格式非预期，CLI 解析失败且未走降级。**排查方法**：确认 CLI 的 `pcm_endpoint` 指向正确的 PCM Core 地址，手动 curl 确认返回格式。后续版本已优化解析异常的降级处理。

**Q：Java 应用出现线程阻塞（BLOCKED on object monitor）如何排查？**
**A：** 现象为线程 dump 中出现 `sun.security.provider.NativePRNG$RandomIO.implNextBytes` 阻塞。原因是 SDK 默认使用 `/dev/random` 阻塞模式获取随机数，系统熵值低（< 100）时线程被卡住。
**解决方案**：升级 SDK 至 `credprovider.plugin >= 1.0.8`；**临时规避**：添加 JVM 参数 `-Djava.security.egd=file:/dev/./urandom`。

**Q：Go SDK 日志文件持续增长未按预期轮转怎么办？**
**A：** 原因是 Go SDK 在 2512 之前版本存在日志轮转 Bug。**解决方案**：升级 Go SDK 至 2512 及以上版本。**临时处理**：使用 `> logfile` 截断日志文件（注意不要 rm 正在写入的文件）。

**Q：PCM Controller 磁盘打满或产生大量日志如何处理？**
**A：** 现象为日志目录 `/home/admin/pcm_controller/logs/api/logs/` 下出现超大文件。
处理方式：
1. 使用 `df -h` 和 `du -sh` 确认磁盘和日志目录使用情况。
2. 清理历史日志文件（保留最近日志）。
3. 排查原因：是否有大量异常请求持续打到 Controller，或定时任务异常导致循环报错。
4. 确认日志轮转配置是否正常。
