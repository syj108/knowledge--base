# 典型问题排查解决方案

本文档涵盖了[[PCM/平台凭证管理服务/index|平台凭证管理服务]]（PCM）及云产品组件在凭证获取、轮转、审计及应急管控等场景下的典型问题排查与解决方案。主要适用场景包括：

1.  **SDK 降级或 Core 异常**：云产品组件获取派生凭证失败、降级使用底表 AK，或 PCM Core 宕机/响应超时。
2.  **派生 AK 队列停止轮转**：派生 AK 队列长时间未创建新 AK，老 AK 未被禁用，轮转暂停。
3.  **热升级凭证异常**：项目热升级后，老凭证未按预期禁用或禁用后导致业务中断。
4.  **异常请求溯源**：收到安全告警或业务反馈存在异常网关请求，需通过事件 ID 或关键字溯源具体调用的 AK。
5.  **AK 调用全量审计**：定期安全审计或凭证轮转前，需盘点底表 AK 在各网关的实际调用情况。
6.  **AK 状态紧急管控**：发生底表 AK 泄露需紧急禁用，或 AK 被误禁用导致业务中断需紧急恢复。
7.  **AK 信息查询**：业务方或安全团队提供账号 ID，需查询其关联的底表 AK 信息。

## 二、排查信息收集

### 1. 必须收集的信息
*   **实例/组件信息**：云产品组件名称、集群名称（ClusterName）、ServerRole 名称。
*   **凭证信息**：底表 AK（initAK）、IAMID（格式：`${CLUSTERNAME}:<serverrole名称>`）、SecretARN、需操作的 AK ID 或 账号 ID（Account ID）。
*   **环境信息**：云版本、部署形态（新部署/热升级）、当前 PCM 管控模式（None/CompatibilityMode/StrictMode/initStrictMode）。
*   **审计与溯源信息**：异常请求的事件 ID 或关键字（如 AK 片段、TraceID）、目标网关代码（如 OSS）、日志查询的时间范围。

### 2. 检查终态的方法
*   **云产品组件侧**：登录组件容器/主机，检查应用日志中 PCM SDK 的返回 code（201/202/203/301/302/401）。
*   **PCM 管控侧**：
    *   登录 PCM Controller 容器，检查派生 AK 队列状态、轮转日志及脚本执行输出状态。
    *   登录 PCM Core 容器，检查共享内存缓存状态及与 Controller 的同步日志。
*   **网关日志与 AK 状态侧**：登录 OPS1 服务检查 `output/` 目录下的扫描结果文件，或通过 PKM 白屏管控平台核对 AK 终态。

### 3. 排查问题步骤与关键结果映射

| 排查结果/日志特征 | 对应场景/原因 |
| :--- | :--- |
| SDK 日志返回 `code=401` | SDK 请求 Core 失败，且本地加密文件无有效凭证，降级返回底表 AK |
| SDK 日志返回 `code=301/302` | SDK 请求 Core 失败，但读取本地加密文件（AES-256-GCM解密）成功，返回本地缓存凭证 |
| SDK 日志返回 `code=201` | 内存缓存命中且未过期，正常返回 |
| SDK 日志返回 `code=202/203` | 内存缓存未命中，从 Core 获取成功，正常返回 |
| Controller 日志显示“最新派生AK保护”触发 | 队列中最早的 AK 是某产品获取的最新 AK，暂停轮转 |
| Controller 日志显示“平台AK访问日志不可行” | 无法确认 AK 是否仍被调用，在第一把队列即将禁用时停止轮转 |
| Core 监控显示节点 Down 或网络 Timeout | PCM Core 宕机或网络异常，触发 SDK 容错降级 |
| 已知异常请求的事件ID/关键字及网关类型 | 场景四：异常网关请求/事件溯源 |
| 需要全量盘点底表AK调用记录 | 场景五：底表AK网关调用情况全量审计 |
| 确认AK泄露需禁用，或业务报错AK被禁用需恢复 | 场景六：底表AK紧急禁用/启用 |
| 仅有账号ID，需获取对应AK | 场景七：根据账号ID查询底表AK |

## 三、解决步骤

### 场景一：SDK 降级返回底表 AK/本地缓存 或 PCM Core 宕机/超时
*   **适用条件**：应用日志显示获取凭证降级（code=401/301/302），或监控告警显示 PCM Core 节点异常、网络不通、响应超时。
*   **实施步骤**：
    1.  **检查 PCM Core 状态**：登录 PCM Core 所在主机/容器，检查 Core 进程是否存活，查看日志确认是否有 OOM 或异常重启。
    2.  **检查网络连通性**：在云产品组件节点执行 `curl` 或 `telnet` 测试到 PCM Core 端点的连通性。
    3.  **检查 PCM Controller 状态**：若 Core 正常，检查 Core 到 Controller 的同步日志，确认 Controller 是否下发最新凭证。登录 Controller 容器检查策略中心服务状态。
    4.  **检查 UMM/AAS 依赖**：若 Controller 正常，检查 Controller 到 UMM（AK管理）和 AAS（账户管理）的调用日志，确认底层账户和 AK 生命周期服务是否正常。
    5.  **应急恢复**：若 PCM 服务短时间无法恢复，且 SDK 缓存丢失导致业务中断，需使用老凭证应急脚本或临时将管控模式降级为 `None` 或 `CompatibilityMode`。
*   **结果验证**：PCM 服务恢复后，观察应用日志，确认 SDK 返回 `code=202/203`，且业务调用恢复正常。

### 场景二：派生 AK 队列停止轮转
*   **适用条件**：PKM 白屏管控界面或 Controller 日志显示队列未轮转，老 AK 未被禁用。
*   **实施步骤**：
    1.  **检查“最新派生 AK 保护”**：登录 PCM Controller，查询该底表 AK 的队列状态。确认是否有某个云产品组件长时间未请求新凭证，导致其持有的 AK 成为“最新”。
        *   *解决*：联系对应云产品团队排查其 SDK 为何未发起新请求（如进程假死、网络隔离），修复后使其获取新 AK，队列将自动恢复轮转。
    2.  **检查“平台 AK 访问日志”状态**：确认平台 AK 访问日志采集是否正常。若日志不可行，系统会在第一把队列即将禁用时停止轮转。
        *   *解决*：排查日志采集链路，恢复 AK 访问日志的可用性。
    3.  **检查 UMM AK 上限**：若队列级别配置为 `ClusterName` 级别，多集群叠加可能导致 UMM 账户的 AK 数量达到上限，无法创建新 AK。
        *   *解决*：在 UMM 中清理废弃 AK，或建议将队列级别修改为推荐的 `initAK` 级别（全局共享）。
*   **结果验证**：触发条件解除后，观察 Controller 日志，确认新派生 AK 创建成功，且过期老 AK 被正常禁用。

### 场景三：热升级后老凭证未按预期禁用或业务中断
*   **适用条件**：项目热升级后，发现老凭证仍在使用，或禁用老凭证后导致业务中断。
*   **实施步骤**：
    1.  **确认当前管控模式**：登录 PKM 白屏管控界面，检查该凭证的 `controlByPcm` 配置。热升级项目默认进入 `CompatibilityMode`（兼容模式），老凭证不会被自动禁用。
    2.  **灰度禁用老凭证**：若需禁用老凭证，切勿一刀切。通过观测平台 AK 访问日志，确认老凭证无流量后，在运维控制台通过灰度变更方式逐步禁用。
    3.  **排查业务中断**：若禁用后业务中断，说明仍有组件在使用老凭证（AK 私用场景或未改造完成）。
        *   *解决*：立即在 PKM 界面恢复老凭证的通用能力（解除禁用），并排查未改造的云产品组件，推动其完成 PCM SDK 接入。
*   **结果验证**：老凭证安全禁用，所有云产品组件均使用派生 AK，业务无报错。

### 场景四：异常网关请求/事件溯源（定位调用AK）
*   **适用条件**：已知异常请求的事件 ID、关键字及所属网关，需定位具体使用的 AK。
*   **实施步骤**：
    1.  登录 OPS1 服务（或可以解析 slsinner 的环境）。
    2.  确保网关日志查询工具（`main`）及配置文件已就绪（配置文件中需填写通过 PCM 控制台手动获取的 SLS 派生 AK/SK 及 Endpoint）。
    3.  执行查询命令：
        ```bash
        ./main query --gateway <网关代码> --keyword "<事件ID或关键字>" --hours <时间范围>
        ```
        *示例：`./main query --gateway OSS --keyword "tzRzgmefjFjXBC4C"`*
*   **结果验证**：查看终端输出的匹配日志，确认发起该请求的 AK 信息。

### 场景五：底表AK网关调用情况全量审计
*   **适用条件**：需要遍历网关中底表 AK 的调用记录，进行安全审计或轮转前评估。
*   **实施步骤**：
    1.  登录 OPS1 服务。
    2.  检查配置文件中的 `scan` 参数（如 `hours_back` 扫描周期，默认 10 小时；`auto_create_index` 是否自动创建索引等）。
    3.  执行全量扫描命令：
        ```bash
        ./main scan
        ```
*   **结果验证**：扫描完成后，检查相对路径 `output/scan_result_{时间戳}.csv`（或 json）文件，确认底表 AK 的调用记录已完整导出。

### 场景六：底表AK紧急禁用/启用（如AK泄露或误禁用恢复）
*   **适用条件**：发生安全事件需紧急禁用指定/全量底表 AK，或业务中断需紧急启用 AK。
*   **实施步骤**：
    1.  登录 PcmController 容器（路径：Product: baseServiceAll → sn: platform-credential-management → sr：PcmController#，进入任意一台容器即可）。
    2.  确认环境变量 `pcm_ctrl_domain` 和 `pcm_rs` 已正确配置（工具会自动从环境变量获取并生成请求签名）。
    3.  根据需求执行对应命令：
        *   禁用指定AK：`python3 manage_ak_status.py disable --ak <AK_ID>`
        *   启用指定AK：`python3 manage_ak_status.py enable --ak <AK_ID>`
        *   禁用全量底表AK：`python3 manage_ak_status.py disable-all`
        *   启用全量底表AK：`python3 manage_ak_status.py enable-all`
*   **结果验证**：观察脚本终端输出，提示“已禁用/已启用”或“启用/禁用完成: X/Y”即为成功。

### 场景七：根据账号ID查询底表AK
*   **适用条件**：业务方或安全团队提供账号 ID（Account ID），需要查询其关联的底表 AK。
*   **实施步骤**：
    1.  登录 PcmController 容器。
    2.  执行查询命令：
        ```bash
        python3 manage_ak_status.py query --account-id <账号ID>
        ```
*   **结果验证**：终端输出“账号 XXX 的 AK: YYY”，确认获取到正确的 AK 信息。若输出“未找到”，则说明该账号无底表 AK。

## 四、非本产品排查

*   **UMM / AAS 服务异常**：若排查确认 PCM Controller 发起创建/获取派生 AK 请求失败，且报错指向 UMM（AK 生命周期管理）或 AAS（账户管理），需联系 **UMM/AAS 产品组** 协助排查底层账户状态及 AK 存储问题。
*   **云产品组件自身问题**：若 PCM 服务正常返回派生 AK（code=202/203），但云产品组件仍报鉴权失败，需联系 **对应云产品组** 排查其业务代码中 AK/SK 的加载逻辑或网关签名校验逻辑。
*   **标准网关鉴权失败 / 网关侧缓存未刷新**：若派生 AK 在 POP、OpenAPI、OSS 等标准网关校验失败，或底表 AK 状态在黑屏工具中修改成功但业务侧仍报鉴权失败，需联系 **网关产品组** 确认网关与 UMM 的同步状态、AK 缓存刷新机制及签名校验逻辑。
*   **SLS 访问凭证失效**：若网关日志查询工具提示 SLS 访问凭证（AK/SK）失效或无权限，需联系 **SLS（日志服务）产品组** 排查派生 AK 状态及授权策略，或检查 PCM 控制台派生 AK 是否过期。
*   **容器环境变量缺失**：若 PcmController 容器无法访问或执行脚本时报错“缺少环境变量 pcm_ctrl_domain 和 pcm_rs”，需联系 **底座/容器平台产品组** 排查容器状态及服务注册变量配置。

## 五、快速定位工具

*   **PKM 白屏管控平台**
    *   **位置**：内部运维控制台 -> 平台凭证管理（PKM）。
    *   **使用方法**：输入底表 AK 或 IAMID，可视化查看凭证当前状态（轮换中/已禁用/正常）、派生 AK 队列详情、管控模式及平台 AK 访问日志关联记录。
*   **PCM 应急降级脚本**
    *   **位置**：PCM 运维工具包（具体路径参考 PCM应急脚本使用说明）。
    *   **使用方法**：当 PCM 服务完全宕机且 SDK 缓存丢失导致业务中断时，在云产品组件节点执行脚本，临时注入老凭证或底表 AK 恢复业务。
*   **日志查询工具 (SLS)**
    *   **位置**：SLS 日志服务 -> PCM 专属 Logstore。
    *   **使用方法**：使用 `SecretARN` 或 `initAK` 作为关键字，检索 SDK、Core、Controller 的全链路调用日志，快速定位请求断点。
*   **网关日志查询工具（CLI）**
    *   **工具名称**：`main`
    *   **运行位置**：OPS1 服务（或可解析 slsinner 的环境）。
    *   **配置依赖**：与 `main` 同目录下的配置文件（需配置 SLS 派生 AK/SK、Endpoint、扫描周期等）。
    *   **常用命令**：`./main -h` 查看帮助，`./main query` 关键字查询，`./main scan` 全量扫描。
*   **底表AK黑屏操作工具（Python脚本）**
    *   **脚本名称**：`manage_ak_status.py`
    *   **运行位置**：PcmController 容器（Product: baseServiceAll → sn: platform-credential-management → sr：PcmController#）。
    *   **环境依赖**：Python3 环境，必须具备环境变量 `pcm_ctrl_domain` 和 `pcm_rs`。
    *   **常用命令**：支持 `enable`、`disable`、`enable-all`、`disable-all`、`query` 操作，通过 `--ak` 或 `--account-id` 传递参数。
