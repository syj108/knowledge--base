# 典型问题排查解决方案

本文档涵盖了[[PCM/平台凭证管理服务/index|平台凭证管理服务]]（PCM）及云产品组件在凭证获取、轮转、审计、应急管控及客户端/服务端异常等场景下的典型问题排查与解决方案。

**问题现象**：业务调用接口失败，日志提示 AK 被禁用、无效或派生失败；监控系统触发 UMMAK 容量告警；客户端产生大量 WARN 日志；Java 应用线程阻塞等。
**适用范围**：PCM 及关联云产品组件，涉及 initAK、底表 AK、派生 AK 的获取、轮转、启用、禁用及容量管理。

## 排查信息收集

### 1. 必须收集的信息
*   **实例/组件信息**：云产品组件名称、集群名称（ClusterName）、ServerRole 名称。
*   **凭证信息**：受影响的 AK ID (`access_id`)、用户 ID (`user_id` / Account ID)、AK 类型（initAK / 底表 AK / 派生 AK）、底表 AK（initAK）、IAMID（格式：`${CLUSTERNAME}:<serverrole名称>`）、SecretARN。
*   **环境与状态信息**：云版本、部署形态（新部署/热升级）、当前 PCM 管控模式（None/CompatibilityMode/StrictMode/initStrictMode）、磁盘使用率、系统熵值。
*   **日志与报错信息**：网关拦截日志、客户端 SDK 日志（含 HTTP 状态码、返回 Msg 及错误 code）、异常请求的事件 ID 或关键字（如 AK 片段、TraceID）、线程 dump 文件。

### 2. 检查终态的方法
*   **云产品组件侧**：登录组件容器/主机，检查应用日志中 PCM SDK 的返回 code（201/202/203/301/302/401）及 HTTP 状态码。
*   **PCM 管控侧**：
    *   登录 PCM Controller 容器，检查派生 AK 队列状态、轮转日志及脚本执行输出状态。
    *   登录 PCM Core 容器，检查共享内存缓存状态、`access.log` 限流状态及与 Controller 的同步日志。
*   **数据库侧**：
    *   **PCM 数据库**：登录 `certificate-lifecycle-manager-server` 服务（db实例: `clm_db`，数据库: `pcm_db`），查询 `init_ak_info` 或 `ak_info` 表确认派生 AK 状态。
    *   **UMMAK 数据库**：登录 `baseService-umm-ak` 服务（db实例: `ummak`，数据库: `ummak`），查询 `accesskey_table` 表确认 `enabled_flag`（1为启用，0为禁用）和 `deleted_flag`。
*   **网关与白屏侧**：登录 OPS1 服务检查扫描结果，或通过 PKM 白屏管控平台核对 AK 终态（控制台仅可查询每个队列最近 14 把派生 AK）。

### 3. 排查问题步骤与关键结果映射

| 排查步骤 / 日志特征 | 关键排查结果 / 对应场景 |
| :--- | :--- |
| 网关报错 AK 被禁用/无效/不存在 | AK 调用网关被拦截（底表 AK 降级或派生 AK 被轮转禁用） |
| SDK 日志返回 `code=401` | SDK 请求 Core 失败，且本地无有效凭证，降级返回底表 AK |
| SDK 日志返回 `code=301/302` | SDK 请求 Core 失败，但读取本地加密文件成功，返回本地缓存凭证 |
| SDK 日志返回 `code=201/202/203` | 内存缓存命中或从 Core 获取成功，正常返回派生 AK |
| 客户端大量刷 `Failed to refresh credential...` WARN | PCM 服务端未部署或不可达，SDK 持续重试失败（已降级，不影响业务） |
| API 返回 HTTP 400/403/502 | 请求参数错误、签名/时钟校验失败或触发 Core 限流 |
| Controller 日志显示“最新派生AK保护”触发 | 队列中最早的 AK 是某产品获取的最新 AK，暂停轮转 |
| Core 监控显示节点 Down 或网络 Timeout | PCM Core 宕机或网络异常，触发 SDK 容错降级 |
| Java 线程 dump 显示 `NativePRNG` 阻塞 | 系统熵值过低导致 `/dev/random` 阻塞 |
| 查询 UMMAK 中 AK 状态 | `accesskey_table` 中 `enabled_flag=0`（AK 被禁用） |
| 查询白屏派生 AK 记录 | 白屏查询不到该 AK（可能超过 14 天被 UMMAK 删除） |
| 查询特定 UID 下 AK 数量 | `accesskey_table` 中某 `user_id` 的 `COUNT(access_id) >= 1000`（触发容量上限） |

## 凭证获取与网关调用异常

### AK 调用网关被拦截 (AK 禁用/无效/不存在)
*   **适用条件**：产品调用网关时报 AK 被禁用、无效或不存在。
*   **实施步骤**：
    1.  从网关日志中取出被拦截的 AK ID。
    2.  在 PCM 控制台查询该 AK 是底表 AK 还是派生 AK。若控制台查不到，进入 `clm_db` 实例切换到 `pcm_db` 执行 SQL 检查：
        ```sql
        use pcm_db;
        select * from ak_info where access_key_id='****';
        ```
    3.  **若为底表 AK 被拦截**：说明 SDK 走了降级逻辑或未适配。先在 PCM 控制台启用该底表 AK 恢复业务（详细操作见“凭证状态管控与应急恢复”），然后查看 SDK 日志 code 确认降级原因。
    4.  **若为派生 AK 被拦截**：说明产品未及时更新到最新派生 AK。尝试重启产品服务刷新 AK 并停止该队列轮转；若无法重启，在控制台手动启用该 AK（详细操作见“凭证状态管控与应急恢复”），并排查 SDK 报错。
*   **结果验证**：网关不再拦截，业务调用恢复正常。

### SDK 降级返回底表 AK/本地缓存 或 PCM Core 宕机/超时
*   **适用条件**：应用日志显示获取凭证降级（code=401/301/302），或监控告警显示 PCM Core 节点异常、网络不通、响应超时。
*   **实施步骤**：
    1.  **检查 PCM Core 状态**：登录 Core 所在主机/容器，检查进程是否存活，查看日志确认是否有 OOM 或异常重启。
    2.  **检查网络连通性**：在云产品组件节点执行 `curl` 或 `telnet` 测试到 PCM Core 端点的连通性。
    3.  **检查 PCM Controller 状态**：若 Core 正常，检查 Core 到 Controller 的同步日志。登录 Controller 容器检查策略中心服务状态。
    4.  **检查 UMM/AAS 依赖**：检查 Controller 到 UMM（AK管理）和 AAS（账户管理）的调用日志，确认底层服务是否正常。
    5.  **应急恢复**：若 PCM 服务短时间无法恢复且 SDK 缓存丢失导致业务中断，需使用老凭证应急脚本或临时将管控模式降级为 `None` 或 `CompatibilityMode`。

### API 调用报错 (HTTP 400/403/502)
*   **适用条件**：SDK 或客户端调用 PCM Core 返回 HTTP 400、403 或 502 错误。
*   **实施步骤**：
    *   **HTTP 400 (请求参数错误)**：根据返回 Msg 定位原因。
        | 返回 Msg | 报错原因 | 排查方向 |
        | --- | --- | --- |
        | `SecretName or x_acs_bearer_token is nil` | SecretName 或 token 为空 | SDK 侧 initakid 和 pcm_endpoint 是否正确 |
        | `SecretName parse fail, SecretName:xxxx` | SecretName 格式错误 | appName 是否正确以 `:` 分隔 |
        | `The access key (AK) is not administered...` | akid 非底表 AK | initakid 是否填写正确的底表 akid |
        | `genJwtKey fail` | 计算 token_key 失败 | Core 内部问题，与 SDK 无关 |
        | `Error in AK rotation led to unsuccessful...` | 请求 Controller 派生失败 | 1. 派生 AK 容量达上限<br>2. IAMID 非法且关闭了非标开关 |
    *   **HTTP 403 (signature error)**：签名 key = sha256(initSK || IKM)，IKM = endpoint 去 https:// → 取域名 → 去掉 -regionid。排查流程如下：
        ```mermaid
        graph TD
            S["Core返回403<br/>reason: signature error"] --> INFO["签名key = sha256(initSK || IKM)<br/>IKM = endpoint去https://→取域名→去掉-regionid"]
            INFO --> Q1{报错范围？}
            Q1 -->|单元region报错<br/>中心region正常| R1["99%是regionid传错<br/>导致两端IKM计算不同"]
            Q1 -->|中心和单元同时报错| R2[initAK/initSK值本身错误]
            Q1 -->|个别产品报错| R3["pcm_endpoint配置错误<br/>（域名拼写/多了路径）"]
            Q1 -->|都确认正确仍403| R4["环境中SK是加密存储的<br/>产品未解密就传给了SDK"]
        ```
    *   **HTTP 403 (nbf 时钟偏差)**：返回 `"nbf" claim not valid until`。新版本 SDK 已增加 5 分钟容错，若仍出现需检查 SDK 所在机器 NTP 同步状态。
    *   **HTTP 502 (限流触发)**：PCM Core 限流基于 IP。检查 `access.log` 中 `limit_req_status` 字段，使用 `tsar -l -i 1 --nginx` 查看 QPS。调整限流配置 `/services/platform-credential-management/user/pcm_conf/pcm_core.json`（单核阈值参考：x86=200r/s, aarch64=189r/s, sw64=80r/s）。

### 客户端产生大量 PCM 相关 WARN 日志
*   **适用条件**：产品日志大量出现 `Failed to refresh credential, pcm server is xxx`，多见于 PCM 服务端未部署或不可达的环境。
*   **实施步骤**：此类 WARN 日志**不影响业务**（SDK 已降级返回原始凭证）。若日志量过大触发告警，可调整客户端日志级别，或在环境中部署 PCM Core 服务。

## 凭证状态管控与应急恢复

> **应急操作优先级原则**：控制台白屏 > 调用接口（容器脚本） > 数据库执行 SQL。当白屏、容器均不可用时，采用数据库操作方案。

### 启用某个已经禁用的 initAK / 底表 AK
*   **适用条件**：确认因为某把特定的 initAK / 底表 AK 被禁用而影响业务。
*   **实施步骤**：
    *   **方法1：白屏操作（优先）**
        *   操作位置：PCM 控制台。
        *   步骤：通过 PCM 控制台的“initAK 管理”功能查询特定 AK，并在操作列中点击“启用”。
    *   **方法2：调用接口（容器脚本）**
        *   操作位置：PcmController 容器内。
        *   步骤：确认环境变量 `pcm_ctrl_domain` 和 `pcm_rs` 已配置，执行：`python3 manage_ak_status.py enable --ak <AK_ID>`（参考 工具 文档）。
    *   **方法3：数据库执行 SQL**
        *   操作位置：UMMAK 数据库（`baseService-umm-ak` / `ummak` / `ummak`）。
        *   步骤：执行以下 SQL 更新状态：
            ```sql
            UPDATE accesskey_table SET enabled_flag=1 WHERE access_id = '{akid}';
            ```
*   **结果验证**：重新查询 `accesskey_table` 确认 `enabled_flag=1`，业务侧重试调用恢复正常。

### 启用全量底表 AK
*   **适用条件**：环境内存在被底表 AK 禁用而影响业务，涉及多把底表 AK 或无法确认具体是哪把底表 AK 被禁用。（注：暂不支持通过白屏解禁全量 AK）
*   **实施步骤**：
    *   **方法1：调用接口（容器脚本）**
        *   操作位置：PcmController 容器内。
        *   步骤：执行 `python3 manage_ak_status.py enable-all`。
    *   **方法2：数据库执行 SQL**
        *   步骤1：进入 PCM 数据库（`clm_db` / `pcm_db`），检索已经禁用的 initAK：
            ```sql
            USE pcm_db;
            SELECT access_key_id FROM init_ak_info WHERE umm_ak_status = 0;
            ```
        *   步骤2：进入 UMMAK 数据库（`ummak`），将步骤1检索到的底表 AK 信息替换到以下 SQL 中执行：
            ```sql
            UPDATE accesskey_table SET enabled_flag=1 WHERE access_id IN ('{akid1}', '{akid2}', '{akid3}');
            ```

### 启用派生 AK（含白屏查询不到的历史 AK）
*   **适用条件**：确认某把派生 AK 被禁用影响业务。注意：白屏仅可查询最近 14 把派生 AK，超过 14 把会在 UMMAK 侧执行删除操作，但 PCM 数据库会保留记录。若白屏未查到，需通过数据库处理。
*   **实施步骤**：
    *   **方法1：白屏操作（针对近14把AK）**
        *   操作位置：PCM 控制台。查询到后通过“启用”操作恢复。
    *   **方法2：数据库执行 SQL（针对白屏查不到或已删除的AK）**
        *   步骤1：进入 PCM 数据库（`pcm_db`）查询派生 AK 记录。
        *   步骤2：如果该 AK 在 UMMAK 中存在但被禁用，进入 `ummak` 数据库直接更新启用状态：
            ```sql
            UPDATE accesskey_table SET enabled_flag=1, hidden_flag=0, deleted_flag=0 WHERE access_id='{akid}';
            ```
        *   步骤3：如果该 AK 在 UMMAK 中已经被删除，则重新创建 AK（需填入对应的 akid、sk 和账号）：
            ```sql
            INSERT INTO `ummak`.`accesskey_table` (`access_id`, `access_key`, `user_id`) VALUES ('{akid}', '{sk}', '{user_id}');
            ```

### 底表 AK 紧急禁用与查询
*   **紧急禁用指定/全量底表 AK**（如 AK 泄露）：在 PcmController 容器执行 `python3 manage_ak_status.py disable --ak <AK_ID>` 或 `python3 manage_ak_status.py disable-all`。
*   **根据账号 ID 查询底表 AK**：在 PcmController 容器执行 `python3 manage_ak_status.py query --account-id <账号ID>`。

## 凭证轮转与容量管理

### 派生 AK 队列停止轮转
*   **适用条件**：PKM 白屏管控界面或 Controller 日志显示队列未轮转，老 AK 未被禁用。
*   **实施步骤**：
    1.  **检查“最新派生 AK 保护”**：确认是否有某个云产品组件长时间未请求新凭证。联系对应团队修复其 SDK 假死或网络隔离问题，获取新 AK 后队列自动恢复。
    2.  **检查“平台 AK 访问日志”状态**：若日志不可行，系统会在第一把队列即将禁用时停止轮转。排查并恢复日志采集链路。
    3.  **检查 UMM AK 上限**：若队列级别为 `ClusterName`，多集群叠加可能导致 AK 数量达上限。在 UMM 中清理废弃 AK，或修改为推荐的 `initAK` 级别。

### UMMAK 容量告警 / 派生失败（单 UID 达到 1000 把上限）
*   **适用条件**：UMMAK 侧每个 `uid` 下最大支持 1000 把有效 AK，当达到 1000 把以后会出现派生失败的情况，或触发容量告警。
*   **实施步骤**：
    1.  **查询特定 UID 下的 AK 数量**（UMMAK 数据库）：
        ```sql
        SELECT user_id, COUNT(access_id) AS access_count FROM accesskey_table WHERE user_id = '{uid}' GROUP BY user_id;
        ```
    2.  **查询是否有 UID 下的 AK 超过 1000 把**：
        ```sql
        SELECT user_id, COUNT(access_id) AS access_count FROM accesskey_table GROUP BY user_id HAVING access_count >= 1000;
        ```
    3.  **清理无用 AK**：分析出环境内已经无用的 AK，在 `ummak` 中置成删除状态：
        ```sql
        UPDATE accesskey_table SET enabled_flag = 0, deleted_flag = 1, modified_time = UNIX_TIMESTAMP() WHERE access_id IN ('{akid1}', '{akid2}');
        ```
*   **结果验证**：确认该 `uid` 下的有效 AK 数量降至 1000 以下，重新触发派生 AK 操作成功。详细处理方案参考 容量问题数据处理 文档。

### 热升级后老凭证未按预期禁用或业务中断
*   **适用条件**：项目热升级后，老凭证仍在使用，或禁用后导致业务中断。
*   **实施步骤**：
    1.  **确认当前管控模式**：热升级项目默认进入 `CompatibilityMode`，老凭证不会被自动禁用。
    2.  **灰度禁用老凭证**：通过观测平台 AK 访问日志确认无流量后，在运维控制台灰度逐步禁用。
    3.  **排查业务中断**：若禁用后中断，说明仍有组件私用老凭证。立即在 PKM 界面恢复老凭证，并推动未改造组件完成 PCM SDK 接入。

## 审计与溯源操作

### 异常网关请求 / 事件溯源（定位调用 AK）
*   **实施步骤**：登录 OPS1 服务，确保网关日志查询工具（`main`）及配置文件就绪。执行：
    ```bash
    ./main query --gateway <网关代码> --keyword "<事件ID或关键字>" --hours <时间范围>
    ```

### 底表 AK 网关调用情况全量审计
*   **实施步骤**：登录 OPS1 服务，检查配置文件中的 `scan` 参数，执行全量扫描：
    ```bash
    ./main scan
    ```
    扫描完成后检查 `output/scan_result_{时间戳}.csv` 文件。

## 客户端 SDK 与工具异常

### Java 应用线程阻塞
*   **适用条件**：线程 dump 出现 `NativePRNG$RandomIO.implNextBytes` 阻塞，系统熵值低（< 100）。
*   **实施步骤**：
    1.  **临时规避**：添加 JVM 参数 `-Djava.security.egd=file:/dev/./urandom`。
    2.  **彻底解决**：升级 SDK 至 `credprovider.plugin >= 1.0.8`。

### Go SDK 日志文件持续增长
*   **适用条件**：Go SDK 版本 < 2512，存在日志轮转 Bug。
*   **实施步骤**：使用 `> logfile` 截断日志文件（**不要** `rm` 正在写入的文件），并升级 Go SDK 至 2512 及以上版本。

### Python SDK RPM 包安装失败
*   **适用条件**：安装 `pcm-python2-sdk-rpm-with-no-six` 报错 `pytz/zoneinfo` 冲突。
*   **实施步骤**：备份旧目录 `mv /home/tops/lib/python2.7/site-packages/pytz /home/tops/lib/python2.7/site-packages/pytz_bak`，然后重新执行 `sudo yum install pcm-python2-sdk-rpm-with-no-six -y`。

### CLI 工具报错 ResponseParseFailure
*   **适用条件**：CLI 返回 `ResponseParseFailure`。
*   **实施步骤**：确认 CLI 的 `pcm_endpoint` 指向正确的 PCM Core 地址，手动 `curl` 确认返回格式，修正配置。

## 服务端与基础设施异常

### PCM Controller 磁盘打满 / 产生大量日志
*   **适用条件**：Controller 日志目录 `/home/admin/pcm_controller/logs/api/logs/` 下出现超大文件。
*   **实施步骤**：使用 `df -h` 和 `du -sh` 确认磁盘状态。清理历史日志文件，确认日志轮转配置正常；排查是否有异常请求或定时任务导致循环报错。

## 非本产品排查

*   **UMM / AAS 服务异常**：若 PCM Controller 发起创建/获取派生 AK 请求失败，且报错指向 UMM 或 AAS，需联系 **UMM/AAS 产品组** 协助排查。
*   **云产品组件自身问题 / 业务侧 SK 加密**：若 PCM 正常返回派生 AK（数据库中 `enabled_flag=1`）但组件仍报鉴权失败，需联系 **对应云产品组** 排查加载逻辑、硬编码错误或配置未更新；若环境中 SK 是加密存储的，需 **业务侧** 自行解密后传入 SDK。
*   **标准网关鉴权失败 / 网关拦截问题**：若 AK 在 PCM 侧状态正常且 SDK 无异常，但在 POP、OSS 等网关校验失败或被拦截，需联系 **网关产品组** 确认网关同步状态、缓存刷新机制及拦截策略。
*   **SLS 访问凭证失效**：若网关日志查询工具提示 SLS 凭证失效，需联系 **SLS 产品组** 排查授权策略或检查派生 AK 是否过期。
*   **容器环境变量缺失**：若 PcmController 容器执行脚本报错缺少 `pcm_ctrl_domain` 等变量，需联系 **底座/容器平台产品组** 排查服务注册变量配置。
*   **NTP 时钟不同步**：若出现 `nbf` 时钟偏差，需联系 **基础设施/运维团队** 排查机器 NTP 同步状态。
*   **时间敏感服务延迟**：接入 PCM 后可能导致部分时间敏感服务延迟加大。可通过设置环境变量 `PCM_TASK_DELAY`（单位 ms，默认 1000ms）调整超时策略，若仍不满足需 **业务侧** 评估架构优化。

## 快速定位工具与命令

*   **PKM 白屏管控平台**：内部运维控制台 -> 平台凭证管理（PKM）。可视化查看凭证状态、派生 AK 队列详情及管控模式。
*   **派生 AK 状态查询 (SQL)**：登录 `clm_db` 实例，切换到 `pcm_db` 数据库执行：
    ```sql
    use pcm_db;
    select * from ak_info where access_key_id='****';
    ```
*   **UMMAK 状态查询 (SQL)**：登录 `ummak` 实例，切换到 `ummak` 数据库执行：
    ```sql
    select * from accesskey_table where access_id='****';
    ```
*   **Core 错误码定位**：检查客户端 SDK 日志中的错误码，参考 Core 错误码快速定位进行快速映射。
*   **PCM 应急降级脚本**：当 PCM 服务完全宕机且 SDK 缓存丢失时，在云产品组件节点执行脚本临时注入老凭证恢复业务。
*   **日志查询工具 (SLS)**：使用 `SecretARN` 或 `initAK` 作为关键字，在 PCM 专属 Logstore 检索全链路调用日志。
*   **网关日志查询工具（CLI）**：在 OPS1 服务运行 `main` 工具。常用命令：`./main query` 关键字查询，`./main scan` 全量扫描。
*   **底表 AK 黑屏操作工具（Python脚本）**：在 PcmController 容器运行 `manage_ak_status.py`。支持 `enable`、`disable`、`enable-all`、`disable-all`、`query` 操作。具体脚本位置及使用方法请参考 工具 文档。
*   **限流排查命令**：在 PCM Core 节点执行 `tsar -l -i 1 --nginx` 查看实时 QPS。
*   **系统熵值检查**：在 Java 应用所在主机执行 `cat /proc/sys/kernel/random/entropy_avail`，若小于 100 则可能触发线程阻塞。
