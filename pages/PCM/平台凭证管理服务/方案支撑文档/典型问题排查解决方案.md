# 典型问题排查解决方案

*   **场景1：云产品组件获取派生凭证失败或降级使用底表AK**
    *   **问题现象**：云产品应用日志中出现获取凭证失败、降级返回底表AK（code=401）或返回本地缓存凭证（code=301/302）的告警或日志。用户反馈业务调用云产品API时出现鉴权失败或使用了非预期的AK。
    *   **适用范围**：所有接入 PCM SDK/CLI 的云产品组件（Java/Go/Python等），涉及 PCM Core、PCM Controller、UMM 等依赖服务。
*   **场景2：派生 AK 队列停止轮转**
    *   **问题现象**：PCM Controller 日志或 PKM 白屏管控界面显示某个底表 AK 的派生 AK 队列长时间未创建新 AK，老 AK 未被禁用，队列轮转处于暂停状态。
    *   **适用范围**：PCM Controller 策略中心，涉及凭证生命周期管理。
*   **场景3：PCM Core 宕机或响应超时**
    *   **问题现象**：SDK 请求 PCM Core 超时，触发 SDK 第二层本地文件读取或降级返回底表 AK。监控告警显示 PCM Core 节点异常或网络不通。
    *   **适用范围**：PCM Core 缓存中间网关及网络链路。
*   **场景4：热升级后老凭证未按预期禁用或业务中断**
    *   **问题现象**：项目热升级后，发现老凭证仍在使用，或禁用老凭证后导致业务中断。
    *   **适用范围**：热升级项目凭证生命周期管理及管控模式切换。
*   **场景5：网关拦截请求并提示 AK 被禁用**
    *   **问题现象**：用户或内部服务在调用云产品API时，请求被网关拦截，返回 401、403 或 500 状态码。错误信息通常包含 `InvalidAccessKeyId`、`Unauthorized`、`AccessKeyId is disabled` 或 `The Access Key is disabled` 等关键字。
    *   **适用范围**：[[PCM/平台凭证管理服务/index|平台凭证管理服务]]（PCM），影响依赖 AK/SK 进行鉴权的各类网关（如 OSS、SLS_INNER、SLS_PUB、ASAPI 等）。

## 二、排查信息收集

*   **必须收集的信息**：
    *   **实例与凭证信息**：云产品组件名称、集群名称（ClusterName）、ServerRole 名称、底表 AK（initAK）、IAMID（格式：`${CLUSTERNAME}:<serverrole名称>`）、SecretARN、报错的 `AccessKeyId`。
    *   **请求与环境信息**：云版本、部署形态（新部署/热升级）、当前 PCM 管控模式（None/CompatibilityMode/StrictMode/initStrictMode）、请求的 `RequestId` 或 `TraceId`（如 `EagleeyeTraceId`）、报错的具体网关/产品组件、报错发生的时间范围、调用方 IP（`ClientIP` / `Source` / `callerIp`）。
*   **检查终态的方法**：
    *   **组件与管控侧**：登录云产品组件容器/主机，检查应用日志中 PCM SDK 的返回 code；登录 PCM Controller/Core 容器检查状态及同步日志；在 PKM 白屏或 PCM 控制台查询凭证当前状态（Enabled/Disabled/轮换中等）。
    *   **网关侧**：登录对应网关的日志系统（如 SLS 日志服务）或登录网关节点查看本地访问日志。
*   **排查问题步骤与关键结果映射**：

| 排查结果/日志特征 | 对应场景/原因 | 日志路径/来源参考 |
| --- | --- | --- |
| SDK 日志返回 `code=401` | SDK 请求 Core 失败，且本地加密文件无有效凭证，降级返回底表 AK | 云产品组件应用日志 |
| SDK 日志返回 `code=301/302` | SDK 请求 Core 失败，但读取本地加密文件（AES-256-GCM解密）成功，返回本地缓存凭证 | 云产品组件应用日志 |
| SDK 日志返回 `code=201/202/203` | 内存缓存命中或从 Core 获取成功，正常返回 | 云产品组件应用日志 |
| Controller 日志显示“最新派生AK保护”触发 | 队列中最早的 AK 是某产品获取的最新 AK，暂停轮转 | PCM Controller 日志 |
| Controller 日志显示“平台AK访问日志不可行” | 无法确认 AK 是否仍被调用，在第一把队列即将禁用时停止轮转 | PCM Controller 日志 |
| Core 监控显示节点 Down 或网络 Timeout | PCM Core 宕机或网络异常，触发 SDK 容错降级 | PCM Core 监控/日志 |
| `error_code: "InvalidAccessKeyId"`, `status: "403"` | 场景5：OSS 网关拦截 | `/apsara/module_logs/oss_tengine/access_log.*` |
| `Status: "401"`, 无明确 ErrorMsg 但鉴权失败 | 场景5：SLS_INNER 网关拦截 | `/apsara/fcgi_agent/ols_operation_2.LOG` |
| `Status: "401"`, `ErrorMsg: "AccessKeyId is disabled: {AK}"` | 场景5：SLS_PUB 网关拦截 | `/apsara/sls/fcgi_agent/ols_operation.LOG` |
| `errorCode: "...accesskeyid.error"`, `errorMessage: "...disabled."` | 场景5：ASAPI 网关拦截 | `/apsara/cloud/data/asapi/.../audit.log` |

## 三、解决步骤

#### 场景一：SDK 降级返回底表 AK / 本地缓存 或 PCM Core 异常
*   **适用条件**：应用日志显示获取凭证降级（code=401/301/302），且 PCM Core 服务异常或网络不通。
*   **实施步骤**：
    1.  **检查 PCM Core 状态**：登录 PCM Core 所在主机/容器，检查 Core 进程是否存活，查看 Core 日志确认是否有 OOM 或异常重启。
    2.  **检查网络连通性**：在云产品组件节点执行 `curl` 或 `telnet` 测试到 PCM Core 端点的连通性。
    3.  **检查 PCM Controller 状态**：若 Core 正常，检查 Core 到 Controller 的同步日志，确认 Controller 是否下发最新凭证。登录 Controller 容器检查策略中心服务状态。
    4.  **检查 UMM/AAS 依赖**：若 Controller 正常，检查 Controller 到 UMM（AK管理）和 AAS（账户管理）的调用日志，确认底层账户和 AK 生命周期服务是否正常。
    5.  **应急恢复**：若 PCM 服务短时间无法恢复，且 SDK 缓存丢失导致业务中断，需使用老凭证应急脚本或临时将管控模式降级为 `None` 或 `CompatibilityMode`。
*   **结果验证**：PCM 服务恢复后，观察应用日志，确认 SDK 返回 `code=202/203`，且业务调用恢复正常。

#### 场景二：派生 AK 队列停止轮转
*   **适用条件**：PKM 白屏管控界面或 Controller 日志显示队列未轮转，老 AK 未被禁用。
*   **实施步骤**：
    1.  **检查“最新派生 AK 保护”**：登录 PCM Controller，查询该底表 AK 的队列状态。确认是否有某个云产品组件长时间未请求新凭证，导致其持有的 AK 成为“最新”。
        *   *解决*：联系对应云产品团队排查其 SDK 为何未发起新请求（如进程假死、网络隔离），修复后使其获取新 AK，队列将自动恢复轮转。
    2.  **检查“平台 AK 访问日志”状态**：确认平台 AK 访问日志采集是否正常。若日志不可行，系统会在第一把队列即将禁用时停止轮转。
        *   *解决*：排查日志采集链路，恢复 AK 访问日志的可用性。
    3.  **检查 UMM AK 上限**：若队列级别配置为 `ClusterName` 级别，多集群叠加可能导致 UMM 账户的 AK 数量达到上限，无法创建新 AK。
        *   *解决*：在 UMM 中清理废弃 AK，或建议将队列级别修改为推荐的 `initAK` 级别（全局共享）。
*   **结果验证**：触发条件解除后，观察 Controller 日志，确认新派生 AK 创建成功，且过期老 AK 被正常禁用。

#### 场景三：热升级后老凭证未按预期禁用或业务中断
*   **适用条件**：项目热升级后，发现老凭证仍在使用，或禁用老凭证后导致业务中断。
*   **实施步骤**：
    1.  **确认当前管控模式**：登录 PKM 白屏管控界面，检查该凭证的 `controlByPcm` 配置。热升级项目默认进入 `CompatibilityMode`（兼容模式），老凭证不会被自动禁用。
    2.  **灰度禁用老凭证**：若需禁用老凭证，切勿一刀切。通过观测平台 AK 访问日志，确认老凭证无流量后，在运维控制台通过灰度变更方式逐步禁用。
    3.  **排查业务中断**：若禁用后业务中断，说明仍有组件在使用老凭证（AK 私用场景或未改造完成）。
        *   *解决*：立即在 PKM 界面恢复老凭证的通用能力（解除禁用），并排查未改造的云产品组件，推动其完成 PCM SDK 接入。
*   **结果验证**：老凭证安全禁用，所有云产品组件均使用派生 AK，业务无报错。

#### 场景四：网关拦截请求并提示 AK 被禁用
*   **适用条件**：OSS、SLS、ASAPI 等网关访问/审计日志中出现 `InvalidAccessKeyId`、`Unauthorized` 或明确提示 `AccessKeyId is disabled`。
*   **实施步骤**：
    1.  **提取关键信息**：从网关日志中提取报错的 `AccessKeyId` 以及 `RequestId` / `TraceId`。
    2.  **核实 PCM 状态**：登录 PCM 管理平台，搜索该 AccessKey ID，确认其当前状态是否为“已禁用”并核实归属及禁用原因。
    3.  **启用 AccessKey**：若确认为误禁用或业务急需恢复，在 PCM 中执行“启用 AccessKey”操作（或通过 PCM OpenAPI 调用启用接口）。
    4.  **处理网关鉴权缓存**：
        *   对于 SLS_INNER / SLS_PUB：若 SLS 侧存在鉴权缓存，可能需要等待缓存过期或联系 SLS 团队刷新缓存。
        *   对于 ASAPI：检查 ASAPI 网关是否需要清理本地鉴权缓存（通常 ASAPI 会定期同步 PCM 状态，若未生效可等待同步周期或重启 ASAPI ApiServer 容器）。
    5.  **通知重试**：通知调用方重新发起请求。
*   **结果验证**：再次发起请求，检查对应网关日志中 `status` 恢复为 `200`（或 `success`），且无拦截报错。

## 四、非本产品排查

*   **UMM / AAS 服务异常**：若排查确认 PCM Controller 发起创建/获取派生 AK 请求失败，且报错指向 UMM（AK 生命周期管理）或 AAS（账户管理），需联系 **UMM/AAS 产品组** 协助排查底层账户状态及 AK 存储问题。
*   **云产品组件自身问题**：若 PCM 服务正常返回派生 AK（code=202/203），但云产品组件仍报鉴权失败，需联系 **对应云产品组** 排查其业务代码中 AK/SK 的加载逻辑或网关签名校验逻辑。
*   **网关缓存未刷新**：若 PCM 中 AK 已启用，但网关（如 ASAPI、SLS）仍拦截并提示 disabled，属于网关侧鉴权缓存未同步问题，需联系 **对应网关产品组（ASAPI/SLS/OSS）** 协助排查缓存刷新机制或手动清理缓存。
*   **签名不匹配（SignatureDoesNotMatch）**：若日志提示签名错误，而非 AK disabled，属于 AK Secret 错误、请求参数被篡改或时间戳不同步，需联系 **调用方/SDK 提供方** 排查签名逻辑，非 PCM 问题。
*   **AK 不存在（AccessKeyIdNotFound）**：若提示 AK 不存在，需确认调用方是否传错 AK，或该 AK 是否已被物理删除。若已删除需重新创建，非 PCM 禁用排查范畴。
*   **标准网关鉴权失败**：若派生 AK 在 POP、OpenAPI 等标准网关校验失败，需联系 **网关产品组** 确认网关与 UMM 的同步状态及签名校验逻辑。

## 五、快速定位工具

*   **PKM 白屏管控平台**：
    *   **位置**：内部运维控制台 -> 平台凭证管理（PKM）。
    *   **使用方法**：输入底表 AK 或 IAMID，可视化查看凭证当前状态（轮换中/已禁用/正常）、派生 AK 队列详情、管控模式及平台 AK 访问日志关联记录。
*   **PCM 应急降级脚本**：
    *   **位置**：PCM 运维工具包（具体路径参考 PCM应急脚本使用说明）。
    *   **使用方法**：当 PCM 服务完全宕机且 SDK 缓存丢失导致业务中断时，在云产品组件节点执行脚本，临时注入老凭证或底表 AK 恢复业务。
*   **日志查询工具与 SLS 查询语句**：
    *   **PCM 全链路日志**：在 SLS 日志服务 -> PCM 专属 Logstore 中，使用 `SecretARN` 或 `initAK` 作为关键字，检索 SDK、Core、Controller 的全链路调用日志，快速定位请求断点。
    *   **ASAPI 网关日志查询**：
        ```sql
        * and errorCode: "asapi.server.request.parameter.accesskeyid.error" and errorMessage: "The Access Key is disabled" | select accessKeyId, callerIp, apiName, time order by time desc limit 10
        ```
    *   **OSS 网关日志查询**：
        ```sql
        * and error_code: "InvalidAccessKeyId" and status: "403" | select access_id, ip, operation, time order by time desc limit 10
        ```
    *   **SLS_PUB 网关日志查询**：
        ```sql
        * and ErrorCode: "Unauthorized" and ErrorMsg: "AccessKeyId is disabled" | select AccessKeyId, ClientIP, ProjectName, microtime order by microtime desc limit 10
        ```
