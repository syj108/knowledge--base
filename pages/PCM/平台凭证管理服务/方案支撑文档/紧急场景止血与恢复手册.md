# 紧急场景止血与恢复手册

在多数异常场景下，PCM 具备完善的自动容错与降级逻辑，无需人工紧急干预即可实现业务“止血”：

| 故障场景 | 系统自动降级行为 | 业务影响 |
| --- | --- | --- |
| 新部署时 PCM Core 还未 ready | SDK 将入参（底表 AK）作为凭证返回 | 无影响（Core 未禁用老 AK） |
| 运行时 PCM Core 宕机 | Core 暂停末期老凭证过期行为；SDK 返回上次获取的老凭证（未在窗口期末尾） | 无影响 |
| 产品独立升级，PCM 未 ready | SDK 将入参作为凭证返回 | 无影响 |
| PCM 和应用均宕机需重拉（SDK 缓存未丢失） | SDK 读取本地加密文件（AES-256-GCM 解密），返回上次获取的老凭证 | 无影响 |
| SDK 请求 Core 失败（网络不通/超时等） | SDK 优先读取本地加密文件缓存；若本地无缓存，则降级返回底表 AK (code=401) | 无影响 |

### 架构级高可用与防误杀保障

* **多级缓存隔离**：PCM SDK 在本地内存、磁盘均设有缓存；PCM Core 具备本地缓存并定时同步 Controller 数据，避免单点故障击穿策略大脑。
* **队列轮转保护**：在禁用队列最早 AK 前，系统会自动触发“产品最新派生 AK 保护”与“平台 AK 访问日志保护”，若发现 AK 仍在使用则自动停止轮转，从源头防止凭证被误杀。
* **变更防误操作机制**：系统默认具备“松→紧变更不自动生效”特性，安全模式从松到紧变更时需 ASO 页面人工确认，可有效防止凭证误禁用等误操作。

## 应急响应基本原则与标准流程

### 操作优先级原则

应急操作优先建议通过控制台白屏操作；当白屏无法访问时，采用在容器中执行脚本（调用服务接口）；当容器无法访问时，直接在数据库中执行 SQL。
**操作优先级：控制台白屏 > 调用接口（容器脚本） > 数据库执行 SQL**

### 标准应急响应与排查流程

当遇到业务访问报错，怀疑是 PCM（[[PCM/平台凭证管理服务/index|平台凭证管理服务]]）禁用 AK（AccessKey）导致，或发生凭证泄露等安全事件时，请按照以下标准流程进行紧急止血与恢复操作：

1. **日志判定与定位**：优先通过各网关的拦截日志进行判定，提取日志中的请求 AK。具体日志特征及查询工具参考下方“异常排查与日志定位”章节。
2. **状态核查**：登录 PCM Controller 提供的白屏管控（PKM）界面或通过 PCM 服务查询提取到的 AK 状态（轮换中/已禁用/正常等），确认是否已被禁用。同时通过日志查询关联能力，检查平台 AK 访问日志，确认该 AK 的实际调用情况。
3. **应急止血处置**：如果确认 AK 存在泄露、异常调用或被误禁用且影响核心业务，立即采用白屏控制台或黑屏操作工具进行禁用止血或重新启用恢复。
4. **根因排查与反馈**：止血完成后，将问题反馈至研发侧，排查 AK 被禁用或泄露的具体原因（如安全策略误杀、凭证泄露自动封禁等），并制定长期修复方案。

### 综合应急处置预案

针对更复杂的灾难场景、大面积凭证失效或全面故障，请严格遵循 PCM应急处置 文档中的标准预案执行一键恢复和应急操作。

## 异常排查与日志定位

### 常见网关拦截日志特征

在应急响应第一步中，需快速识别不同网关的拦截日志特征：

**OSS 网关拦截**
* **核心特征**：`"error_code": "InvalidAccessKeyId"`, `"status": "403"`
* **日志示例**：
```json
{
  "error_code": "InvalidAccessKeyId",
  "status": "403",
  "access_id": "5hN1RkUhRn43iNfw",
  "request_id": "69EB1A0A3E6DA93539F3A4CE",
  "operation": "GetBucketAcl",
  "host": "cn-wulan-env17e-d01-as-console-cdn.oss-cn-wulan-env17e-d01-a.intra.env17e.shuguang.com"
}
```

**SLS 网关拦截 (SLS_INNER / SLSPUB)**
* **核心特征**：`"Status": "401"`, `"ErrorCode": "Unauthorized"`, `"ErrorMsg": "AccessKeyId is disabled: <AK_ID>"`
* **日志示例 (SLSPUB)**：
```json
{
  "Status": "401",
  "AccessKeyId": "Khz7a1kmKMZDCBXj",
  "ErrorCode": "Unauthorized",
  "ErrorMsg": "AccessKeyId is disabled: Khz7a1kmKMZDCBXj",
  "Method": "ListShards",
  "ProjectName": "ali-cdsslshybridcluster-a-20260323-015f-sls-admin"
}
```

**ASAPI 网关拦截**
* **核心特征**：`"errorCode": "asapi.server.request.parameter.accesskeyid.error"`, `"errorMessage": "The specified AccessKey ID (<AK_ID>) is invalid. Details: (The Access Key is disabled.)."`
* **日志示例**：
```json
{
  "errorCode": "asapi.server.request.parameter.accesskeyid.error",
  "errorMessage": "The specified AccessKey ID (VidKjhddRaas4tMA) is invalid. Details: (The Access Key is disabled.).",
  "errorSuggestion": "Check whether the AccessKey pair exists and is enabled.",
  "accessKeyId": "VidKjhddRaas4tMA",
  "apiName": "ListAllLevel1Orgs",
  "status": "failed"
}
```

### 网关日志查询工具使用

在发生凭证泄露或异常调用等紧急场景时，需使用**网关日志查询工具**快速定位异常 AK。请确保工具配置文件（包含 SLS 凭证及 Endpoint）已正确配置并与 CLI 工具置于同目录下。

* **根据事件ID精准定位AK**
  ```bash
  ./main query --gateway <网关代码> --keyword "<事件ID>"
  ```
  *示例：`./main query --gateway OSS --keyword "tzRzgmefjFjXBC4C"`*

* **全量扫描底表AK调用记录**
  扫描结果将自动输出至相对路径的 `output/scan_result_{时间戳}.csv` 文件中：
  ```bash
  ./main scan
  ```

### 服务端异常日志排查与定位

当 PCM 服务出现异常或请求失败时，需登录 `PCM Core` 所在的两个 Docker 容器进行日志排查（**注意：两个 Docker 均需查询**）。更多详细排查思路请参考 PCM排查思路&常见问题。

**1. Error 日志排查（确认是否 PCM-Core 报错返回）**
* 有具体 RequestID：
  ```bash
  grep -rn "<requestid>" /opt/tengine/logs/error.<date>.log
  ```
* 无具体 RequestID（按 AKID、IAMID 和时间段复合筛选）：
  ```bash
  grep "<akid>" /opt/tengine/logs/error.<date>.log | grep "<iamid>" | awk '$1 >= "<start_date>" && $2 >= "<start_time>" && $2 <= "<end_time>"'
  ```

**2. Access 日志排查（确认 PCM-Core 是否接收到请求）**
* 有具体 RequestID：
  ```bash
  grep -rn "<requestid>" /opt/tengine/logs/access.<date>.log
  ```
* 无具体 RequestID（结合 `time_local` 和 AKID 筛选）：
  ```bash
  grep -E '"time_local": "(<date>:<hour>:<minute>|...)"' /opt/tengine/logs/access.<date>.log | grep "<akid>"
  ```

**Access 日志核心参数说明：**

| 参数名称 | 参数含义 |
| --- | --- |
| remote_addr | 请求源地址 |
| Gateway-POP-Tunnel-ID | tunnel-id |
| X-Aliyun-Vpc-Id | vpc-id |
| remote_port | 请求端口 |
| time_local | 请求完成的时间 |
| request_uri | 请求的 uri，包含 iamid、secretname、endpoint 等信息 |
| request_method | 请求方法 |
| status | HTTP 返回码 |
| http_user_agent | 请求代理客户端信息 |
| request_time | Tengine 收到请求到发完响应的总耗时 |
| SecretName | secretname，包含 initakid 和 pcm_endpoint 信息 |
| IamId | 表示请求服务身份，对应 SDK 填写的 appname，当 HTTP 报错时可能会为空 |
| x_acs_bearer_token | 请求发送 JWT |
| x_sdk_client | PCM-SDK 版本 |
| limit_req_status | 限流状态，未限流显示 "PASSED"，限流显示 "-" |
| eagleeye_traceid | 即 requestid，可根据此查询对应 error_log 是否有错误日志 |

## 凭证异常应急恢复与止血操作

### 单个底表 AK (initAK) 禁用恢复

* **适用场景**：产品调用网关时报 AK 被禁用/无效/不存在，且经排查确认使用的是底表 AK（SDK 未成功获取派生 AK 走了降级逻辑，或未适配使用底表 AK）。
* **白屏操作**：通过 PCM 控制台（白屏）的“底表 AK 管理”模块查询特定 AK 的禁用状态，并在操作中启用该 AK，优先恢复业务。
  > *注意：白屏控制台不提供底表 AK 禁用能力，禁用操作需严格参考相关变更文档执行。*
* **调用接口（黑屏脚本）**：当白屏不可用时，在容器中执行工具脚本进行恢复：
  ```bash
  python <黑屏操作脚本>.py enable --ak <AK_ID>
  ```
* **数据库操作**：当白屏、容器均不可用时，进入 UMMAK 数据库（Service: `baseService-umm-ak`，DB实例: `ummak`，数据库: `ummak`），执行以下 SQL：
  ```sql
  update accesskey_table set enabled_flag=1 where access_id = '{akid}';
  ```
* **后续排查**：检查 SDK 日志 code，确认是哪种降级场景，参考 Core 错误码快速定位进行根因分析。

### 全量底表 AK 禁用恢复

* **适用场景**：环境内存在被底表 AK 禁用而影响业务，涉及多把底表 AK 或无法确认具体某把底表 AK。
  > **注意**：暂不支持通过白屏解禁全量 AK。
* **调用接口（黑屏脚本）**：在容器中执行工具脚本进行恢复：
  ```bash
  python <黑屏操作脚本>.py enable-all
  ```
* **数据库操作**：当容器不可访问时采用此方案。
  1. **获取全量底表 AK**：进入 PCM 托管的底表 AK 数据库（Service: `certificate-lifecycle-manager-server`，DB实例: `clm_db`，数据库: `pcm_db`）。
     ```sql
     use pcm_db;
     select access_key_id from init_ak_info where umm_ak_status = 0;
     ```
  2. **启用全量底表 AK**：进入 UMMAK 数据库（Service: `baseService-umm-ak`，DB实例: `ummak`，数据库: `ummak`），将 `access_id` 参数替换为上一步检索到的底表 AK 信息并执行：
     ```sql
     update accesskey_table set enabled_flag=1 where access_id in ('ak1','ak2','ak3');
     ```

### 派生 AK 禁用恢复

* **适用场景**：产品已使用派生 AK，但该派生 AK 已被轮转禁用，产品未及时更新到最新的派生 AK。
  > **注意**：每个派生队列中通过白屏仅可查询最近 14 把派生 AK。超过 14 把会在 UMMAK 侧执行删除操作，但 PCM 数据库会保留记录。若白屏未查到，可能是 14 天前派生的 AK，需通过 PCM 数据库查询。
* **优先恢复（重启）**：重启产品服务。重启通常会触发 AK 刷新从而恢复可用，随后在控制台停止该队列的轮转。
* **白屏操作**：通过白屏查询派生 AK，查询后执行启用操作恢复。
* **调用接口（黑屏脚本）**：若业务无法重启，使用黑屏工具启用：
  ```bash
  python <黑屏操作脚本>.py enable --ak <AK_ID>
  ```
* **数据库操作**：
  1. **查询派生 AK**：进入 PCM 数据库（Service: `certificate-lifecycle-manager-server`，DB实例: `clm_db`，数据库: `pcm_db`）。
     ```sql
     use pcm_db;
     ```
  2. **在 UMMAK 中启用**（Service: `baseService-umm-ak`，DB实例: `ummak`，数据库: `ummak`）：
     * **若 AK 存在**，直接更新启用状态：
       ```sql
       update accesskey_table set enabled_flag=1,hidden_flag=0,deleted_flag=0 where access_id='{akid}';
       ```
     * **若 AK 已被删除**，重新创建 AK（需替换 `access_id`、`access_key`、`user_id`）：
       ```sql
       INSERT INTO `ummak`.`accesskey_table` (`access_id`, `access_key`, `user_id`) VALUES ('{akid}', '{sk}', '{uid}');
       ```
* **后续排查**：检查 SDK 是否仅获取一次而未持续轮转，结合 SDK 报错信息排查。

### 手动创建临时派生 AK 应急

* **适用场景**：当某个应用需要使用临时 AK 登录，或者使用的 initAK 被禁用且无法立即恢复时，可通过创建临时派生 AK 进行应急止血。
* **操作步骤**：
  1. 进入 ASO 控制台（安全管理 -> 账户安全 -> 平台凭据管理 PCM），切换到“派生 AK 管理”标签页，点击“创建临时 AK”按钮。
  2. 填写相关申请信息：
     * `initAKID`：托管到 PCM 的基线或底表 AK（需与所使用账号的原始 AK 对应）。
     * `申请者ID (IAMID)`：服务的身份标识（常规为 `集群 + : + sr` 拼接，如 `StandardCloudCluster-A-20250906-00bf:PcmController`。若系统提示已存在，可在末尾拼接任意字符串）。
     * `有效天数`：限制在 1~365 天。
     * `申请者类型`：选择 ApsaraStackProduct 或 Other。
     * 选填信息：CloudID、ProductName、ClusterName、ServiceName（建议准确填写以便追溯使用方）。
  3. 创建成功后，**立即复制并保存 AK 和 SK 明文**。
     > *注意：SK 明文仅在创建成功弹窗内展示一次，关闭后系统不再显示。若不慎关闭弹窗，需重新创建临时 AK。*

### 紧急止血操作（禁用异常AK）

确认异常 AK 后，需立即使用**底表AK黑屏操作工具**进行止血。操作前请确保运行环境中已正确配置环境变量 `pcm_ctrl_domain`（PCM控制节点域名）和 `pcm_rs`（签名密钥）。

* **禁用指定AK（常规止血）**：`python <黑屏操作脚本>.py disable --ak <AK_ID>`
* **一键禁用全量AK（极端灾难止血）**：`python <黑屏操作脚本>.py disable-all`
* **辅助查询账号AK**：`python <黑屏操作脚本>.py query --account-id <账号ID>`

### AK 容量告警与派生失败恢复

* **适用场景**：UMMAK 侧每个 UID 下最大支持 1000 把有效 AK，达到上限后会出现派生失败的情况（容量告警）。详细处理可参考容量问题数据处理。
* **查询容量**：进入 UMMAK 数据库。
  * 检查特定 UID 下的 AK 数量：
    ```sql
    SELECT user_id, COUNT(access_id) AS access_count FROM accesskey_table where user_id = '{uid}' GROUP BY user_id;
    ```
  * 查询是否有 UID 下的 AK 超过 1000 把：
    ```sql
    SELECT user_id, COUNT(access_id) AS access_count FROM accesskey_table GROUP BY user_id HAVING access_count >= 1000;
    ```
* **清理无用 AK**：分析出环境内已经无用的 AK，在 UMMAK 中将其置为删除状态以释放容量：
  ```sql
  update accesskey_table set enabled_flag = 0, deleted_flag = 1 , modified_time = UNIX_TIMESTAMP() where access_id in ('{akid1}', '{akid2}');
  ```

## 架构与配置级风险应急

### 半轮转模式首次获取失败
* **场景**：部分产品采用半自动轮转模式（仅启动时获取一次派生 AK），若首次获取失败（如限流、网络抖动），产品将持续使用底表 AK 或无有效凭据运行，且不会自动恢复。
* **应急操作**：重启产品服务以重新触发派生 AK 获取。若 PCM 服务端异常，临时在控制台启用底表 AK 保障业务运行。

### 底表禁用导致重启服务无凭据
* **场景**：底表 AK 被 PCM 禁用后，若此时 PCM 链路（Core + Controller）不可用，重启的服务将无法获取任何有效凭据（底表已禁、派生获取失败、本地无缓存），导致业务直接中断。
* **应急操作**：立即在 PCM 控制台临时启用底表 AK，确保重启服务能获取凭据。随后紧急排查并恢复 PCM 服务端链路，待 PCM 服务完全恢复且派生 AK 正常下发后，再次禁用底表 AK。

### AK 轮转状态停止应急排查
若在“AK 申请详情”中发现轮转状态已停止，需排查以下原因并进行相应处理：
1. 检查 IAMID 中是否包含 `CLOSE_AUTO_ROTATE` 状态（表示该队列默认不轮转）。
2. 确认使用该产品的队列中，是否有产品未及时更新（参考 平台凭证管理服务（PCM）介绍）。
3. 确认使用该队列的产品中，是否有产品仍在使用第 7 把 AK。

## 客户端与 SDK 异常应急处理

### Java 应用线程阻塞
* **场景**：线程 dump 中出现 `BLOCKED` 状态，堆栈指向 `NativePRNG$RandomIO.implNextBytes` 和 `PcmSecretCredentialManager.persistCredentials`。原因是系统熵值低（< 100）时，SDK 默认使用 `/dev/random` 阻塞模式获取随机数导致线程卡死。
* **应急操作**：
  1. **临时规避**：在 JVM 启动参数中添加 `-Djava.security.egd=file:/dev/./urandom` 并重启应用。
  2. **彻底解决**：升级 SDK 至 `credprovider.plugin >= 1.0.8` 版本。

### Go SDK 日志文件持续增长导致磁盘打满
* **场景**：Go SDK 产生的日志文件不断增大，未按预期轮转（2512 之前版本存在 Bug）。
* **应急操作**：
  1. **临时止血**：使用 `> logfile` 命令截断日志文件释放空间（**严禁使用 `rm` 删除正在写入的文件**）。
  2. **彻底解决**：升级 Go SDK 至 2512 及以上版本。

### CLI 工具报错 ResponseParseFailure
* **场景**：CLI 工具返回 `ResponseParseFailure`，原因是 `pcm_endpoint` 地址配置错误，响应格式非预期且未走降级。
* **应急操作**：确认 CLI 的 `pcm_endpoint` 指向正确的 PCM Core 地址，并手动使用 `curl` 测试该地址确认返回格式正确。升级 CLI 工具至最新版本以支持异常降级。

### 已知缺陷与修复版本速查表

| 问题现象 | 修复版本 | 潜在风险 | 应急建议 |
| --- | --- | --- | --- |
| CLI 服务端返回异常不降级（ResponseParseFailure） | 2025-12-23 更新版本 | CLI 直接不可用 | 升级 CLI 工具或修正 `pcm_endpoint` 配置 |
| Java SDK 线程阻塞（`/dev/random` 熵值问题） | `credprovider.plugin >= 1.0.8` | 应用线程卡死 | 添加 JVM 参数 `-Djava.security.egd=file:/dev/./urandom` |
| Go SDK 日志文件不轮转 | SDK >= 2512 版本 | 磁盘打满 | 使用 `> logfile` 截断日志并升级 SDK |

## 服务端与基础设施应急处理

### PCM Controller 磁盘打满
* **场景**：Controller 日志目录 `/home/admin/pcm_controller/logs/api/logs/` 下出现超大文件，导致磁盘空间不足。
* **应急操作**：
  1. 使用 `df -h` 确认磁盘使用情况，使用 `du -sh /home/admin/pcm_controller/logs/api/logs/` 查看日志目录大小。
  2. **清理日志**：清理历史日志文件，务必保留最近日志。
  3. **排查根因**：检查是否有大量异常请求持续打到 Controller，或定时任务异常导致循环报错，并确认日志轮转配置是否正常。

### Core 限流导致 HTTP 502
* **场景**：SDK 或客户端大量收到 HTTP 502 响应，大概率是触发了 PCM Core 基于 IP 的限流策略（同一台机器多组件可能互相误伤）。
* **应急操作**：
  1. 检查 `access.log` 中的 `limit_req_status` 字段确认是否被限流。
  2. 使用 `tsar -l -i 1 --nginx` 查看实时 QPS。
  3. **调整限流**：修改限流配置文件 `/services/platform-credential-management/user/pcm_conf/pcm_core.json`。单核阈值参考：x86=200r/s, aarch64=189r/s, sw64=80r/s。

## 极端灾难场景与人工恢复

### PCM 与应用同时宕机且 SDK 本地缓存丢失
* **故障现象**：应用重启后无法从内存或本地磁盘获取缓存凭证，且 PCM 服务不可用，导致业务完全中断。
* **应急止血操作**：
  1. **首选方案**：紧急排查并恢复 PCM 服务（包括 PCM Core 与 PCM Controller），待服务 Ready 后应用自动获取新派生凭证。
  2. **兜底方案**：若 PCM 短时间内无法恢复，需立即使用**老凭证应急脚本**，将原始底表 AK（initAK）直接配置到应用中，临时绕过 PCM 托管以恢复业务。

### 热升级后老凭证需紧急回退
* **故障现象**：项目热升级接入 PCM 后，新派生凭证存在兼容性问题，需紧急回退至老凭证。
* **应急止血操作**：
  1. 热升级项目默认**不禁用**原始凭证的通用能力，老凭证依然可用。
  2. 若需彻底回退，可通过观测日志在运维控制台进行灰度操作，取消老凭证的禁用计划，确保应用平滑降级至使用原始底表 AK。
