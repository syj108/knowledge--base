# 紧急场景止血与恢复手册

在平台凭证管理服务（PCM）运行过程中，针对各类组件异常或升级场景，系统设计了完善的容错与止血机制。以下为常见灾难场景下的应急表现与恢复策略：

| 灾难场景 | 系统/SDK 应急行为 | 业务影响 | 恢复与止血操作 |
| --- | --- | --- | --- |
| 新部署时 PCM Core 还未 ready | SDK 将入参（原始底表 AK）作为凭证返回 | 无影响（Core 未禁用老 AK） | 等待 PCM Core 启动就绪即可自动恢复 |
| 运行时 PCM Core 宕机 | SDK 返回上次获取的老凭证（未在窗口期末尾） | 无影响 | Core 宕机后末期过期老凭证行为暂停，重启 Core 后自动恢复同步 |
| 产品独立升级，PCM 未 ready | SDK 将入参（原始底表 AK）作为凭证返回 | 无影响 | 等待 PCM 服务就绪，无需人工干预 |
| PCM 和应用都挂了需重拉（SDK 缓存未丢失） | SDK 读取本地加密文件，返回上次获取的老凭证 | 无影响 | 应用重启后自动读取本地缓存，待 PCM 恢复后自动续期 |
| **PCM 和应用都挂了需重拉（SDK 缓存丢失）** | **无法自动降级，业务中断** | **业务中断** | **需优先恢复 PCM 服务，或使用老凭证应急脚本手动注入底表 AK 进行止血** |

## 凭证获取异常降级机制

当云产品组件通过 PCM SDK/CLI 获取凭证发生网络异常或服务端故障时，SDK 具备多级容错降级能力，确保业务不中断：

1. **第一层降级（内存与本地文件缓存）**：当 SDK 请求 PCM Core 失败（如网络不通、超时或 Core 异常）时，SDK 会自动读取本地加密文件（AES-256-GCM 解密）。若本地文件存在有效凭证，则直接返回本地缓存凭证。
2. **第二层降级（底表 AK 兜底）**：若本地缓存文件也丢失或无效，且 PCM 初始化服务异常或报错，SDK 将直接降级返回入参中的原始底表 AK（initAK），保障应用能够继续使用基础凭证进行调用。

## 凭证轮转异常与防误删保护

在凭证生命周期管理中，为防止因轮转策略导致正在使用的凭证被误禁用（即“止血”防误操作），PCM Controller 提供了以下轮转保护机制：

* **产品最新派生 AK 保护**：在准备禁用队列中最早的 AK 时，系统会校验该 AK 是否为某产品获取的“最新”派生 AK。若是，则暂停队列轮转，直到该产品获取了更新的 AK，防止业务因凭证被禁而中断。
* **平台 AK 访问日志不可行保护**：当日志系统不可行，PCM 无法确认即将禁用的派生 AK 是否仍被调用时，系统将在第一把队列即将禁用时自动停止轮转。
* **平台 AK 访问日志可信保护**：在禁用派生 AK 前，系统会检查平台 AK 访问日志。若日志显示该 AK 当前仍有网关调用记录，则立即停止轮转。

## 运维管控与灰度恢复操作

针对存量改造、热升级及模式切换等高风险运维场景，PCM 提供了白屏化与灰度止血手段：

* **灰度禁用老凭证**：在热升级场景下，原始凭证的通用能力**不会被自动禁用**。如需禁用老凭证，必须通过观测日志并在运维控制台（PKM）以灰度变更的方式逐步进行，严禁一刀切操作。
* **模式变更防误操作**：当管控模式从“松”向“紧”变更时（如切换至 StrictMode），变更**不会自动生效**。系统会在 ASO 页面提示人工确认处理，防止因误操作导致大面积凭证失效。
* **白屏管控（PKM）**：在紧急情况下，运维人员可通过 PKM 可视化界面快速查询凭证状态、关联 AK 使用日志，并手动干预凭证的轮换与禁用状态，降低应急操作门槛。
