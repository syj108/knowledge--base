# 紧急场景止血与恢复手册

对于大部分常规异常场景，PCM SDK 具备自动容错降级能力，不会对业务造成影响：

| 故障场景 | SDK 降级行为 | 业务影响 |
| --- | --- | --- |
| 新部署时 PCM Core 还未 ready | 将入参作为返回 | 无影响（Core 未禁用老 AK） |
| 运行时 PCM Core 挂了 | 返回上次获取的老凭证（未在窗口期末尾） | 无影响 |
| 产品独立升级，PCM 未 ready | 将入参作为返回 | 无影响 |
| PCM 和应用都挂了需重拉（SDK 缓存未丢失） | 返回上次获取的老凭证 | 无影响 |

### 伴随告警与误报判断
在触发上述降级机制期间，客户端可能会产生大量 PCM 相关的 WARN 日志（如 `Failed to refresh credential, pcm server is xxx`）。
* **应急判断**：此类日志**不影响业务**（SDK 已降级返回原始凭证），主要影响是触发客户端告警监控。
* **处理建议**：无需紧急止血，可直接忽略或调整告警规则。

## 灾难场景应急操作（SDK 缓存丢失）

当发生 **PCM 和应用都挂了需重拉，且 SDK 缓存丢失** 的极端灾难场景时，SDK 无法自动降级，将导致**业务中断**。

**紧急止血与恢复操作：**
* **恢复 PCM 服务**：优先排查并紧急恢复 PCM 服务端，使应用重新具备获取派生凭证的能力。
* **使用老凭证应急脚本**：若 PCM 服务短时间内无法恢复，需立即使用**老凭证应急脚本**进行止血，保障业务快速恢复。

## AK 调用网关被拦截应急恢复

这是 PCM 接入后最核心的排查场景，产品调用网关时可能报 AK 被禁用/AK 无效/AK 不存在。首先需从网关日志中取出被拦截的 AK ID，在控制台查询是底表 AK 还是派生 AK。

### 底表 AK 被拦截
* **场景判断**：产品在使用底表 AK，说明 SDK 没有成功获取派生 AK，走了降级逻辑或使用底表 AK 未适配。
* **应急恢复**：在 PCM 控制台**启用该底表 AK**，立即恢复业务。
* **后续排查**：查 SDK 日志 code，确认是哪种降级场景。

### 派生 AK 被拦截
* **场景判断**：产品已经在使用派生 AK，但这把派生 AK 已被轮转禁用。
* **应急恢复**：
  1. 通常**重启服务**会刷新 AK 导致可用，然后停止该队列的轮转。
  2. 若无法重启服务，需要**手动启用 AK**（参见 [《PCM应急处置》](https://alidocs.dingtalk.com/i/nodes/MNDoBb60VLYDGNPytBomBqkPJlemrZQ3?utm_scene=team_space&iframeQuery=anchorId%3Duu_mogmd4kosy5jbbqysjf)）。
* **后续排查**：最可能原因为仅获取一次，未持续轮转。

## 磁盘与日志异常应急处理

### PCM Controller 磁盘打满
* **故障现象**：Controller 日志目录 `/home/admin/pcm_controller/logs/api/logs/` 下出现超大文件，磁盘空间不足。
* **应急操作**：
  1. 确认磁盘使用情况：`df -h`
  2. 查看日志目录大小：`du -sh /home/admin/pcm_controller/logs/api/logs/`
  3. **清理历史日志文件**（保留最近日志）
  4. 确认日志轮转配置是否正常
* **后续排查**：排查产生大量日志的原因（是否有大量异常请求持续打到 Controller，是否有定时任务异常导致循环报错）。

### Go SDK 日志文件持续增长
* **故障现象**：Go SDK 产生的日志文件不断增大，未按预期轮转（2512 之前版本存在日志轮转 Bug）。
* **应急操作**：使用 `> logfile` **截断日志文件**（**注意：绝对不要 rm 正在写入的文件**）。
* **彻底解决**：升级 Go SDK 至 2512 及以上版本。

## 应用与组件异常应急处理

### Java 应用线程阻塞
* **故障现象**：线程 dump 中出现阻塞堆栈（如 `sun.security.provider.NativePRNG$RandomIO.implNextBytes`），原因为 SDK 默认使用 `/dev/random` 阻塞模式获取随机数，系统熵值低（< 100）时线程被卡住。
* **应急操作**：添加 JVM 参数 `-Djava.security.egd=file:/dev/./urandom` 进行临时规避。
* **彻底解决**：升级 SDK 至 `credprovider.plugin >= 1.0.8`。

### Python SDK RPM 包安装失败
* **故障现象**：安装 `pcm-python2-sdk-rpm-with-no-six` 报错（关键字：`pytz/zoneinfo`、`cpio: File from package already exists as a directory`），原因为系统已有 `/home/tops/lib/python2.7/site-packages/pytz/` 目录与 RPM 包冲突。
* **应急操作**：
  ```bash
  mv /home/tops/lib/python2.7/site-packages/pytz /home/tops/lib/python2.7/site-packages/pytz_bak
  sudo yum install pcm-python2-sdk-rpm-with-no-six -y
  ```

### Core 限流导致 HTTP 502
* **故障现象**：Core 返回 502，大概率触发限流。
* **应急操作**：
  1. 检查 access.log 中 `limit_req_status` 字段。
  2. 使用 `tsar -l -i 1 --nginx` 查看 QPS。
  3. **调整限流配置**文件：`/services/platform-credential-management/user/pcm_conf/pcm_core.json`。
  4. 阈值参考（单核）：x86=200r/s, aarch64=189r/s, sw64=80r/s。
