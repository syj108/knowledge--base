# 紧急场景止血与恢复手册

- **初始化异常降级**：PCM 初始化服务异常或报错时，SDK 会将入参（原始底表 AK）作为凭证返回。
- **缓存命中降级**：如果存在缓存，SDK 将返回最近一次从服务端获取的凭证。
- **请求失败多级降级**：当 SDK 请求 PCM Core 失败（如网络不通、超时或 Core 异常）时，触发以下降级路径：
  - 读取本地加密文件（AES-256-GCM 解密）。
  - 若本地文件有有效凭证，返回本地缓存凭证。
  - 若本地文件也没有有效凭证，最终降级返回底表 AK。

## Core 降级保护机制
- **宕机保护**：PCM Core 宕机后，末期过期老凭证的禁用行为将暂停。SDK 会返回上次获得的老凭证（未在窗口期末尾），业务依然可以正常使用该凭证。
- **压力缓解**：Core 作为中间缓存网关，避免所有 SDK 请求直接打到 Controller，防止策略大脑被击穿导致全局故障。

## 灾难场景应急操作与恢复指南

### 应急操作基本原则
应急操作优先建议控制台白屏操作，当白屏无法访问时，采用在容器中执行脚本（调用服务接口），当容器无法访问时，直接在数据库中执行 SQL。
**优先级：控制台白屏 > 调用接口（容器脚本） > 数据库执行 SQL**

### 常见故障场景与自动恢复
在以下常见故障场景中，系统可通过自动降级实现止血，业务无感知或无影响：

| 故障场景 | SDK 自动止血行为 | 业务影响 |
| --- | --- | --- |
| 新部署时 PCM Core 还未 ready | 将入参（底表 AK）作为返回 | 无影响（Core 未禁用老 AK） |
| 运行时 PCM Core 挂了 | 返回上次获取的老凭证（未在窗口期末尾） | 无影响 |
| 产品独立升级，PCM 未 ready | 将入参作为返回 | 无影响 |
| PCM 和应用都挂了需重拉（SDK 缓存未丢失） | 返回上次获取的老凭证 | 无影响 |
| 客户端产生大量 PCM 相关 WARN 日志（如 `Failed to refresh credential`） | SDK 降级返回原始凭证，不影响业务调用 | 无影响（可能触发客户端告警监控；无服务端时 SDK 会持续尝试连接产生日志） |

### 网关拦截日志排查与定位

### AK 调用网关被拦截（AK被禁用）应急恢复
当产品调用网关时报 AK 被禁用/AK 无效/AK 不存在时，需快速判断被拦截的 AK 类型并执行恢复：

#### 底表 AK 被拦截
- **核心判断**：产品在使用底表 AK，说明 SDK 没有成功获取派生 AK（走了降级逻辑）或未适配使用底表 AK。
- **应急恢复**：
  1. **优先恢复业务**：在 PCM 控制台启用该底表 AK。
     ![底表AK判定](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/oJGq75k4LxgGzlAK/img/2fd7cf34-2df5-4f17-9e5e-693a440fa80f.png)
  2. **后续排查**：业务恢复后，检查 SDK 日志 code，确认是哪种降级场景导致未拿到派生 AK。

#### 派生 AK 被拦截
- **核心判断**：产品已经在使用派生 AK，但这把派生 AK 已被轮转禁用（最可能原因为仅获取一次，未持续轮转）。
- **应急恢复**：
  1. **重启服务刷新**：通常重启服务会刷新 AK 导致可用，然后停止该队列的轮转。
     ![停止轮转](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/oJGq75k4LxgGzlAK/img/d4de6c7b-0ccf-4072-97a2-f3076bedea7c.png)
  2. **手动启用 AK**：若无法重启服务，需通过控制台或数据库手动启用该派生 AK（参考下文“启用派生 AK”操作）。

### 极端灾难场景恢复操作（SDK 缓存丢失）
当发生 **PCM 和应用都挂了需重拉，且 SDK 缓存丢失** 的极端灾难场景时，系统将无法自动降级，会导致**业务中断**。

**应急恢复操作**：
1. **优先恢复 PCM 服务**：紧急排查并恢复 PCM Core 及 Controller 服务，使 SDK 能够重新获取派生凭证。
2. **使用老凭证应急脚本**：若 PCM 服务短时间内无法恢复，需立即执行老凭证应急脚本，使用原始底表 AK 临时恢复业务调用，确保业务止血。

### 底表禁用联动风险提示
- **风险说明**：底表 AK 被 PCM 禁用后，产品的凭据供给完全依赖 PCM 链路（Core + Controller）。
- **影响评估**：对于本地有缓存的运行中服务暂时无影响；但若此时 PCM 不可用且服务发生重启，将因“底表已禁、派生获取失败、本地无缓存”而拿不到任何有效凭据，导致业务直接中断。
- **操作建议**：在执行底表 AK 禁用操作前，需充分评估环境内 PCM 服务的可用性，避免引发大面积业务中断。

## 组件异常与资源耗尽应急止血

### PCM Controller 磁盘打满
- **现象**：Controller 日志目录 `/home/admin/pcm_controller/logs/api/logs/` 下出现超大文件，磁盘空间不足。
- **应急处理**：
  1. 确认磁盘使用情况：`df -h`
  2. 查看日志目录大小：`du -sh /home/admin/pcm_controller/logs/api/logs/`
  3. 清理历史日志文件（保留最近日志）。
  4. 排查产生大量日志的原因（是否有大量异常请求持续打到 Controller，或定时任务异常导致循环报错），并确认日志轮转配置是否正常。

### Go SDK 日志文件持续增长
- **现象**：Go SDK 产生的日志文件不断增大，未按预期轮转（2512 之前版本存在日志轮转 Bug）。
- **应急处理**：使用 `> logfile` 截断日志文件（**注意**：不要 `rm` 正在写入的文件）。
- **根本解决**：升级 Go SDK 至 2512 及以上版本。

### Java 应用线程阻塞
- **现象**：线程 dump 中出现阻塞堆栈（如 `NativePRNG$RandomIO.implNextBytes`），SDK 默认使用 `/dev/random` 阻塞模式获取随机数，系统熵值低（< 100）时线程被卡住。
- **应急处理**：临时规避可添加 JVM 参数 `-Djava.security.egd=file:/dev/./urandom`。
- **根本解决**：升级 SDK 至 `credprovider.plugin >= 1.0.8`。

## 凭证启用与恢复操作 SOP

### 启用单个 initAK
- **适用场景**：确认因为某把 AK 被禁用而影响业务。
- **白屏操作**：通过 PCM 控制台的 initAK 管理功能查询特定 AK，并在操作中启用该 AK。
  ![initAK白屏启用](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/Lk3lbmbxZoXdJOm9/img/71c6499e-968c-4b82-aa25-e3093bc7d9b7.png)
- **调用接口（容器中执行脚本）**：当白屏不可用时采用此方案。可登录 PcmController 容器，使用**底表 AK 黑屏操作工具**执行 `python3 manage_ak_status.py enable --ak {akid}`（详见下文“底表 AK 黑屏操作工具”章节），或参考 [《工具》](https://alidocs.dingtalk.com/i/nodes/7NkDwLng8Za7QYkeHxdzN0A7JKMEvZBY?utm_scene=team_space&iframeQuery=anchorId%3Duu_mocpgly2iwborsrkk7e)。
- **数据库操作**：当白屏、容器均不可用时采用此方案。
  1. 进入 UMMAK 数据库：
     - Service：`baseService-umm-ak`
     - DB 实例：`ummak`
     - 数据库：`ummak`
  2. 执行 SQL 启用 AK：
     ```sql
     update accesskey_table set enabled_flag=1 where access_id = '{akid}';
     ```

### 启用全量底表 AK
- **适用场景**：环境内存在被底表 AK 禁用而影响业务，涉及多把底表 AK 或无法确认某把底表 AK 时采用。
- **注意**：暂不支持通过白屏解禁全量 AK。
- **调用接口（容器中执行脚本）**：当白屏不可用时采用此方案。可登录 PcmController 容器，使用**底表 AK 黑屏操作工具**执行 `python3 manage_ak_status.py enable-all`（详见下文“底表 AK 黑屏操作工具”章节），或参考 [《工具》](https://alidocs.dingtalk.com/i/nodes/7NkDwLng8Za7QYkeHxdzN0A7JKMEvZBY?utm_scene=team_space&iframeQuery=anchorId%3Duu_mocpgly2iwborsrkk7e)。
- **数据库操作**：当容器不可访问时采用此方案。
  1. 获取全量底表 AK（PCM 托管的底表 AK 存储在 `clm_db` 实例的 `pcm_db` 数据库中）：
     - Service：`certificate-lifecycle-manager-server`
     - DB 实例：`clm_db`
     - 数据库：`pcm_db`
     - 切换数据库：
       ```sql
       use pcm_db;
       ```
     - 检索已经禁用的 initAK：
       ```sql
       select access_key_id from init_ak_info where umm_ak_status = 0;
       ```
  2. 启用全量底表 AK（在 UMMAK 数据库中执行）：
     - Service：`baseService-umm-ak`
     - DB 实例：`ummak`
     - 数据库：`ummak`
     - 执行 SQL（执行前将 `access_id` 字段参数改成步骤 1 中检索到的底表 AK 信息）：
       ```sql
       update accesskey_table set enabled_flag=1 where access_id in ('ak1','ak2','ak3');
       ```

### 启用派生 AK
- **适用场景**：确认某把派生 AK 被禁用影响业务。
- **白屏操作**：白屏支持查询派生 AK，查询后可通过启用操作恢复。
  ![派生AK白屏启用](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/Lk3lbmbxZoXdJOm9/img/610ebe22-d389-44fe-a31b-cce2b4c221c2.png)
  > **注意**：每个派生队列中通过白屏仅可以查询最近 14 把派生 AK。如果超过 14 把，会在 UMMAK 侧执行删除操作，但 PCM 数据库会保留派生 AK 记录。当通过白屏未查询到该 AK 时，可能是 14 天前派生的 AK，可通过 PCM 数据库进行查询。
- **数据库操作**：
  1. 查询派生 AK（PCM 数据库）：
     - Service：`certificate-lifecycle-manager-server`
     - DB 实例：`clm_db`
     - 数据库：`pcm_db`
     - 切换数据库：
       ```sql
       use pcm_db;
       ```
  2. 在 UMMAK 中启用：
     - 如果 AK 存在，直接更新启用状态：
       ```sql
       update accesskey_table set enabled_flag=1, hidden_flag=0, deleted_flag=0 where access_id='{akid}';
       ```
     - 如果 AK 已经删除，重新创建 AK（需替换 `access_id`、`access_key`、`user_id`）：
       ```sql
       INSERT INTO `ummak`.`accesskey_table` (`access_id`, `access_key`, `user_id`) VALUES ('{akid}', '{sk}', '{uid}');
       ```

## 容量告警场景处理

### 告警说明
UMMAK 侧每个 UID 下最大支持 1000 把有效 AK，当达到 1000 把以后会出现派生失败的情况（注：目前主要在测试环境出现，生产现场暂未出现）。详细数据处理可参考 [《容量问题数据处理》](https://alidocs.dingtalk.com/i/nodes/QG53mjyd800agdlKHbek2aXQ86zbX04v)。

### 查询与排查
- **检查特定 UID 下的 AK 数量**：
  ```sql
  SELECT user_id, COUNT(access_id) AS access_count 
  FROM accesskey_table 
  WHERE user_id = '{uid}' 
  GROUP BY user_id;
  ```
- **查询是否有 UID 下的 AK 超过 1000 把**：
  ```sql
  SELECT user_id, COUNT(access_id) AS access_count 
  FROM accesskey_table 
  GROUP BY user_id 
  HAVING access_count >= 1000;
  ```

### 清理无用 AK
分析出环境内已经无用的 AK，在 UMMAK 中置成删除状态：
```sql
update accesskey_table 
set enabled_flag = 0, deleted_flag = 1, modified_time = UNIX_TIMESTAMP() 
where access_id in ('{akid1}', '{akid2}');
```

## 运维管控与应急辅助工具

### 管控与辅助能力概述
在紧急场景下，可通过 PCM Controller 提供的管控能力进行应急操作和状态确认：
- **灰度禁用老凭证**：支持热升级后以运维变更方式逐步禁用老凭证，而非一刀切，防止因误操作导致大面积业务故障。
- **松→紧变更不自动生效**：管控模式从松到紧变更时不自动生效，需 ASO 页面提示人工处理，防止误操作引发紧急故障。
- **白屏管控（PKM）**：提供可视化的凭证管理界面，降低运维门槛，便于在紧急情况下快速查看凭证状态并进行人工干预。
- **日志查询关联**：提供日志查询能力，关联 AK 使用记录，帮助运维人员在应急场景下快速判断凭证是否仍在被调用，从而决策是否可以安全禁用。

### 网关日志查询工具
用于在应急场景下排查 AK 使用情况，支持通过网关和事件 ID 查询日志详细信息，以及在网关日志中扫描底表 AK 使用情况。
