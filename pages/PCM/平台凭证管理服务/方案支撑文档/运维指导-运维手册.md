# 运维指导-运维手册

当前源内容中未提供 PCM 服务自身的具体数据库与表结构信息。PCM 主要作为凭证缓存网关与策略中心，其底层凭证存储与账户管理依赖以下服务：
*   **UMM（AK 生命周期管理）**：负责 AK 的存储与生命周期管理，接收 Controller 指令执行凭证轮换和禁用操作。
*   **AAS（账户管理服务）**：负责平台账户统一管理，与 UMM 联动形成账户-凭证关联关系。

## 关键日志与监控

当前源内容中未提供具体的日志文件路径与轮转策略。运维过程中主要关注以下日志与监控能力：
*   **平台 AK 访问日志**：用于检查底表 AK 和派生 AK 是否在网关中有调用记录。PCM Controller 会关联此日志，判断即将禁用的派生 AK 是否仍有产品在调用，从而触发队列轮转保护机制。
*   **PCM Controller 日志查询**：提供日志查询能力，关联 AK 使用记录，辅助判断是否可以安全禁用老凭证。

## 问题排查 SOP 与通用场景排查思路

基于 PCM 的高可用与容错逻辑，以下为通用异常场景的排查思路与系统降级行为：

| 异常场景 | 系统行为与排查思路 | 业务影响 |
| --- | --- | --- |
| **新部署时 PCM Core 还未 ready** | **行为**：SDK 将入参（底表 AK）作为返回。<br>**排查**：确认 PCM Core 组件启动状态与就绪探针。 | 无影响（Core 未禁用老 AK） |
| **运行时 PCM Core 宕机** | **行为**：SDK 返回上次获取的老凭证（未在窗口期末尾）。<br>**排查**：检查 Core 组件健康状态、Pod 状态及网络连通性。 | 无影响 |
| **产品独立升级，PCM 未 ready** | **行为**：SDK 将入参作为返回。<br>**排查**：确认 PCM 服务整体状态。 | 无影响 |
| **PCM 和应用都挂了需重拉（SDK 缓存未丢失）** | **行为**：SDK 读取本地加密文件（AES-256-GCM 解密），返回上次获取的老凭证。<br>**排查**：检查本地缓存文件完整性。 | 无影响 |
| **PCM 和应用都挂了需重拉（SDK 缓存丢失）** | **行为**：降级返回底表 AK（code=401）。<br>**排查**：**需先恢复 PCM 或使用老凭证应急脚本**，检查 SDK 本地持久化目录是否被误清理。 | **业务中断** |
| **SDK 请求 Core 失败 (网络不通/超时/Core异常)** | **行为**：触发第二层容错，读取本地加密文件；若无有效凭证，降级返回底表 AK。<br>**排查**：检查 SDK 到 Core 的网络策略、Core 服务负载及超时配置。 | 视本地缓存情况而定 |

## 版本升级指南与管控策略

**热升级兼容策略**
*   **新部署项目**：根据 `restrict` 取值禁用原始通用能力，应用使用凭证进入定时轮换状态。
*   **热升级项目**：原始凭证**不禁用**其通用能力，进入定时轮换状态；如需禁用老凭证，需通过观测日志在运维控制台（PKM 白屏管控）灰度进行，避免一刀切。
*   **非 PCM 托管凭证**：一切照旧；若使用了 PCM SDK/CLI 但未被托管，将入参 initAK 返回让应用接着使用。

**管控模式变更指南**
PCM 支持多种管控模式（None、CompatibilityMode、StrictMode、initStrictMode）。
*   **松→紧变更不自动生效**：当管控模式从松到紧变更时，系统不会自动生效，需在 ASO 页面根据提示进行人工处理，以防止误操作导致业务中断。
*   **灰度禁用**：支持热升级后以运维变更方式逐步禁用老凭证。
