# 业务逻辑时序图

[[PCM/平台凭证管理服务/index|平台凭证管理服务]]（PCM）的核心机制主要围绕**凭证获取与多级缓存**、**异常容错降级**以及**后台凭证轮转**三个维度展开。其核心业务逻辑涵盖了**应用侧通过 SDK 获取派生 AK 流程**、**运维管理侧的手动创建临时 AK 流程**。同时，针对业务运行中可能出现的 AK 被禁用、UMMAK 容量告警等异常情况，PCM 制定了标准的排查与处置闭环流程（包含严格的启用降级策略），并配套提供了**网关日志查询**与**底表 AK 黑屏操作**等运维工具，以支持日常管理与应急处置。

## 核心凭证获取与轮转全局时序

以下为完整的[[DDoS/DDoS基础防护/产品对内文档/业务逻辑时序图|业务逻辑时序图]]，展示了凭证获取的多级缓存机制、异常降级路径以及后台异步流转过程：

```mermaid
sequenceDiagram
    autonumber
    participant App as 云产品组件<br/>(Java/Go/Python/CLI)
    participant SDK as PCM SDK/CLI<br/>(本地凭证管理)
    participant Core as PCM Core<br/>(凭证缓存网关)
    participant Ctrl as PCM Controller<br/>(策略中心)
    participant UMM as UMM<br/>(AK管理)
    participant AAS as AAS<br/>(账户管理)

    rect rgb(240, 248, 255)
    Note over App, AAS: 正常获取派生凭证流程 (多级缓存机制)
    App->>SDK: 请求凭证 (传入底表AK/SK + IAMID + SecretARN)
    SDK->>SDK: 第一层：检查内存缓存 (凭据是否存在且未过期)
    
    alt 内存缓存命中且未过期
        SDK-->>App: 返回缓存凭证 (code=201)
    else 内存缓存未命中或已过期
        SDK->>Core: 请求凭证 (SecretName + PcmToken JWT签名)
        Core->>Core: 检查共享内存缓存 (initak_queue/endpoint_queue等)
        
        alt Core缓存命中
            Core-->>SDK: 返回缓存的派生凭证
        else Core缓存未命中或已过期
            Core->>Ctrl: 请求派生新凭证 (initAK + IAMID + endpoint)
            Ctrl->>UMM: 创建/获取派生AK
            UMM->>AAS: 关联账户凭证
            AAS-->>UMM: 返回账户数据
            UMM-->>Ctrl: 返回派生AK
            Ctrl-->>Core: 返回派生凭证
            Core->>Core: 写入共享内存缓存
            Core-->>SDK: 返回新派生凭证
        end
        
        SDK->>SDK: 更新内存缓存 + 加密持久化到本地文件
        SDK-->>App: 返回派生凭证 (code=202/203)
    end
    end

    rect rgb(255, 245, 238)
    Note over App, AAS: 异常降级路径 (SDK请求Core失败/超时/异常)
    SDK->>SDK: 第二层：读取本地加密文件 (AES-256-GCM解密)
    alt 本地文件有有效凭证
        SDK-->>App: 返回本地缓存凭证 (code=301/302)
    else 本地文件无有效凭证
        SDK-->>App: 降级返回底表AK (code=401)
    end
    end

    rect rgb(245, 255, 245)
    Note over Core, AAS: 后台异步任务与轮转机制
    Core-->>Ctrl: 定时同步 Controller (保持共享缓存最新)
    Ctrl-->>UMM: 定期轮转派生AK (创建新AK、禁用老化凭证)
    end
```

**时序图核心流程说明：**

*   **正常获取凭证流程（多级缓存机制）**
    *   **L1 内存缓存**：云产品组件通过 PCM SDK/CLI 请求凭证时，SDK 优先检查本地内存缓存。若命中且未过期，直接返回（Code 201），实现最低延迟。
    *   **L2 共享缓存**：若本地未命中，SDK 携带 JWT 签名向 PCM Core 发起请求。Core 检查共享内存缓存，若命中则返回，SDK 更新本地缓存并加密持久化到磁盘。
    *   **L3 策略与底层创建**：若 Core 缓存也未命中或已过期，Core 将请求转发至 PCM Controller。Controller 调用 UMM 和 AAS 创建或获取新的派生 AK，逐层返回并更新各级缓存（Code 202/203）。
*   **异常降级路径（高可用容错）**
    *   当 SDK 请求 PCM Core 失败（如网络不通、超过默认 1s 超时时间、同 IP 高频请求触发限流返回 502、签名或时钟校验失败返回 403、参数错误 400 或 Core 宕机）时，自动触发降级逻辑。此过程会产生大量 WARN 级别日志。
    *   SDK 读取本地磁盘的加密文件，若存在有效凭证则返回本地缓存（Code 301/302）。
    *   若本地文件也无有效凭证（如首次启动且 Core 不可用），则最终降级返回原始底表 AK（Code 401），确保业务不中断。
*   **后台异步流转（定时同步与轮转）**
    *   **缓存同步**：PCM Core 定时与 PCM Controller 同步，保持共享缓存数据最新，避免所有 SDK 请求直接击穿 Controller。
    *   **凭证轮转**：标准模式下，SDK 会在后台定时向 Core 请求刷新派生 AK，确保凭证持续有效。PCM Controller 定期执行派生 AK 队列轮转，创建新 AK 并禁用老化凭证。轮转过程受“最新派生 AK 保护”和“平台 AK 访问日志保护”等机制约束，确保正在使用中的凭证不会被误禁用。
    *   **半轮转模式风险提示**：对于部分采用“半轮转模式”的产品（仅在启动时获取一次凭证），若首次获取恰好失败（如网络抖动、Core 限流），将导致后续持续使用底表 AK 或无有效凭据运行，且不会自动恢复，存在潜在的业务中断风险。

## SDK经网关获取派生AK详细时序

本章节是对全局时序中 SDK 到 Core 链路的细化，补充了网关（Tengine）转发以及 PCM Core 针对 `secretARN` 的 12 小时缓存机制细节。对于持续使用派生 AK 的产品，理论上每 12 小时会触发一次底层查询并记录一条 AK 申请日志。

```mermaid
sequenceDiagram
    autonumber
    participant App as 应用客户端(PCM SDK)
    participant Gateway as 网关(Tengine)
    participant Core as PCM Core
    participant Cache as 本地缓存(12h)
    participant DB as 底层存储

    App->>Gateway: 发起获取派生AK请求(携带IamId, SecretName, x_sdk_client等)
    Gateway->>Gateway: 记录 access 日志(包含请求源、耗时、状态等)
    Gateway->>Core: 转发请求至 PCM Core
    Core->>Core: 解析请求参数并校验身份(IamId)与限流状态
    Core->>Cache: 查询底表 secretARN 缓存
    
    alt 缓存命中
        Cache-->>Core: 返回 secretARN
    else 缓存未命中或已过期
        Core->>DB: 查询底层存储获取 secretARN
        DB-->>Core: 返回 secretARN
        Core->>Cache: 更新本地缓存(有效期12小时)
        Core->>Core: 记录 AK申请日志
    end
    
    Core->>Core: 生成/获取派生AK凭证
    Core-->>Gateway: 返回派生AK/SK及处理状态
    Gateway-->>App: 返回最终响应结果
    
    opt 若处理过程中发生异常
        Core->>Core: 记录 error 日志(包含 eagleeye_traceid 等排查信息)
    end
```

## 控制台手动创建临时AK流程

适用于某个应用需要使用临时 AK 登录，或使用的 initAK 被禁用时的应急/临时处理场景。该流程展示了运维用户通过 PCM 控制台与系统交互创建临时凭证的过程。需要注意的是，SK 明文仅在创建成功后的弹窗中展示一次，系统不对外提供 SK 明文信息的二次查询能力。

```mermaid
sequenceDiagram
    autonumber
    actor User as 运维用户
    participant Console as PCM控制台
    participant Core as PCM Core
    participant DB as 底层存储

    User->>Console: 进入派生AK管理页，点击“创建临时AK”
    User->>Console: 输入申请者(IAMID)、initAKID、有效天数、原因等信息
    Console->>Core: 提交创建临时AK请求
    Core->>Core: 校验参数合法性(initAKID、有效天数1~365天等)
    Core->>DB: 查询initAKID对应的底表AK状态及信息
    DB-->>Core: 返回底表AK信息
    Core->>Core: 生成临时AK与SK明文
    Core->>DB: 持久化派生AK申请记录
    Core-->>Console: 返回临时AK与SK明文
    Console-->>User: 弹窗展示AK/SK明文(提示仅展示一次)
    User->>User: 复制并妥善保存AK/SK
```

## AK禁用排查与处置流程

当业务系统出现访问报错时，若怀疑是 PCM 禁用 AK 导致，需按照以下标准排查与处置时序流程进行操作。该流程涵盖了从网关日志分析、PCM 状态确认到应急处置及研发反馈的完整闭环。

```mermaid
sequenceDiagram
    participant Client as 客户端/业务系统
    participant Gateway as 业务网关(OSS/SLS/ASAPI等)
    participant Ops as 运维排查人员
    participant PCM as PCM凭证服务
    participant RD as 研发团队

    Client->>Gateway: 发起业务请求(携带AccessKey)
    Gateway-->>Client: 拒绝访问并返回错误状态码(如403/401/500)
    Gateway->>Gateway: 生成拦截日志(记录AK、ErrorCode等特征)
    
    Note over Ops: 发现业务报错，怀疑AK被PCM禁用
    Ops->>Gateway: 检索并查看网关拦截日志
    Gateway-->>Ops: 返回日志详情(如OSS的InvalidAccessKeyId或SLS的Unauthorized)
    Ops->>Ops: 提取日志中的请求AK
    
    Ops->>PCM: 查询目标AK的当前状态
    PCM-->>Ops: 返回AK状态(确认已禁用)
    
    Note over Ops: 确认因AK禁用导致网关拦截
    Ops->>Ops: 执行应急处置方案(如重新启用AK或更换有效凭证)
    Ops->>RD: 反馈问题至研发侧，要求排查禁用原因
    RD-->>Ops: 接收反馈并介入排查根本原因
```

**流程说明：**

1.  **日志判定与拦截分析**：不同网关的拦截日志特征不同（例如 OSS 返回 `403 InvalidAccessKeyId`，SLS 返回 `401 Unauthorized`，ASAPI 返回 `AccessKey is disabled` 等），排查时需优先通过对应网关的日志提取请求 AK。当发生拦截时，需区分 AK 类型进行针对性排查：
    *   **底表 AK 被拦截**：说明 SDK 未能成功获取派生 AK 从而走了降级逻辑，或产品未适配 PCM。需优先在控制台启用底表 AK 恢复业务，再通过 SDK 日志错误码排查降级原因。
    *   **派生 AK 被拦截**：说明产品已使用派生 AK，但该 AK 已被轮转禁用。通常是因为产品未持续轮转（如半轮转模式仅获取一次），或轮转刷新失败。需重启服务刷新 AK 或手动启用 AK 进行应急恢复。
2.  **状态核实**：提取 AK 后，必须通过 PCM 服务二次确认该 AK 的真实状态，避免误判。
3.  **闭环处置**：确认禁用后，先执行应急处置恢复业务（具体的 AK 启用降级操作策略详见下文 **AK启用应急处置降级时序**），随后务必反馈研发侧排查 AK 被禁用的根本原因，防止问题复发。

## AK启用应急处置降级时序

平台凭证管理服务(PCM)在进行AK（initAK、底表AK、派生AK）启用等应急操作时，遵循严格的降级策略：**控制台白屏 > 调用接口（容器脚本） > 数据库执行SQL**。以下为核心应急处置业务的时序流转图：

```mermaid
sequenceDiagram
    autonumber
    participant Op as 运维人员
    participant Console as PCM控制台(白屏)
    participant Container as 容器/脚本(接口)
    participant PCM_DB as PCM数据库(pcm_db)
    participant UMM_DB as UMMAK数据库(ummak)

    Note over Op, UMM_DB: 场景：启用被禁用的AK (initAK/底表AK/派生AK)
    
    rect rgb(240, 248, 255)
    Note right of Op: 优先级 1：白屏操作
    Op->>Console: 1. 尝试登录PCM控制台并操作启用AK
    alt 白屏可用
        Console->>UMM_DB: 调用后端接口更新 enabled_flag=1
        UMM_DB-->>Console: 更新成功
        Console-->>Op: 返回启用成功
    else 白屏不可用
        Console-->>Op: 访问失败/不可用
    end
    end

    rect rgb(255, 245, 238)
    Note right of Op: 优先级 2：调用接口(容器脚本)
    alt 白屏不可用
        Op->>Container: 2. 登录容器执行应急工具脚本
        alt 容器可用
            Container->>UMM_DB: 调用服务接口更新 AK 状态
            UMM_DB-->>Container: 更新成功
            Container-->>Op: 脚本执行成功
        else 容器不可用
            Container-->>Op: 容器无法访问
        end
    end
    end

    rect rgb(245, 255, 245)
    Note right of Op: 优先级 3：数据库直接操作 (SQL)
    alt 容器不可用或需处理全量/历史派生AK
        Op->>PCM_DB: 3. (可选) 查询全量底表AK或14天前的历史派生AK记录
        PCM_DB-->>Op: 返回 AK 列表 (access_id)
        Op->>UMM_DB: 4. 执行 SQL 更新 accesskey_table (enabled_flag=1)
        Note right of UMM_DB: 注：派生AK若在UMMAK侧已被物理删除<br/>则需执行 INSERT 语句重新创建AK记录
        UMM_DB-->>Op: SQL 执行成功，AK 恢复启用
    end
    end
```

## 容量告警处理时序

当UMMAK侧单个UID下有效AK数量达到上限（最大1000把）时，会导致派生AK失败并可能触发容量告警。以下为容量告警场景的数据排查与清理时序图：

```mermaid
sequenceDiagram
    autonumber
    participant Op as 运维人员
    participant Monitor as 监控/告警系统
    participant UMM_DB as UMMAK数据库(ummak)

    Note over Op, UMM_DB: 场景：UMMAK 容量告警处理 (单UID下AK>=1000导致派生失败)
    
    Monitor->>Op: 1. 触发容量告警或派生失败异常
    Op->>UMM_DB: 2. 查询特定 UID 下的 AK 数量 (GROUP BY user_id)
    UMM_DB-->>Op: 返回 access_count
    
    alt access_count >= 1000
        Op->>Op: 3. 分析环境内已无用的 AK 列表
        Op->>UMM_DB: 4. 执行 SQL 清理无用 AK<br/>(enabled_flag=0, deleted_flag=1)
        UMM_DB-->>Op: 清理成功，释放容量额度
        Op->>Monitor: 5. 确认派生功能恢复正常
    else access_count < 1000
        Op->>Op: 排查其他导致派生失败的原因
    end
```

## 网关日志查询工具业务时序

网关日志查询工具主要通过与 SLS（日志服务）交互，完成网关日志的拉取、底表 AK 使用情况的扫描解析以及结果的本地持久化。

```mermaid
sequenceDiagram
    participant 用户
    participant 网关日志查询工具
    participant SLS日志服务
    participant 本地文件系统

    用户->>网关日志查询工具: 执行查询/扫描命令 (scan/query)
    网关日志查询工具->>网关日志查询工具: 读取配置文件 (SLS凭证、Endpoint、扫描周期等)
    
    alt 扫描模式 (scan)
        网关日志查询工具->>SLS日志服务: 请求指定时间范围的网关日志
        SLS日志服务-->>网关日志查询工具: 返回分页日志数据
        网关日志查询工具->>网关日志查询工具: 解析日志并扫描底表AK使用情况
        网关日志查询工具->>本地文件系统: 写入扫描结果 (CSV/JSON等格式)
        本地文件系统-->>网关日志查询工具: 写入成功
    else 查询模式 (query)
        网关日志查询工具->>SLS日志服务: 根据网关代码和关键字查询日志
        SLS日志服务-->>网关日志查询工具: 返回匹配的日志详情
        网关日志查询工具->>网关日志查询工具: 格式化日志信息
    end
    
    网关日志查询工具-->>用户: 输出执行结果或本地文件路径
```

## 底表AK黑屏操作工具业务时序

底表 AK 黑屏操作工具主要通过与 PCM Controller 服务交互，完成 AK 状态的查询与变更。所有 API 请求均需携带基于环境变量动态生成的 MD5 签名 Headers 以通过服务端的安全校验。

```mermaid
sequenceDiagram
    participant 用户
    participant AK黑屏操作工具
    participant PCM_Controller服务

    用户->>AK黑屏操作工具: 执行AK管理命令 (如 disable-all / enable / query)
    AK黑屏操作工具->>AK黑屏操作工具: 读取环境变量 (pcm_ctrl_domain, pcm_rs)
    AK黑屏操作工具->>AK黑屏操作工具: 生成MD5请求签名及鉴权Headers
    
    alt 全量操作 (enable-all / disable-all)
        AK黑屏操作工具->>PCM_Controller服务: 查询底表AK列表 (/initAK/queryInitAkList)
        PCM_Controller服务-->>AK黑屏操作工具: 返回全量AK列表数据
        loop 遍历AK列表
            AK黑屏操作工具->>PCM_Controller服务: 更新AK状态 (/operation/updateAkStatus)
            PCM_Controller服务-->>AK黑屏操作工具: 返回单个AK更新结果
        end
        AK黑屏操作工具->>AK黑屏操作工具: 统计成功与失败数量
    else 单个AK操作 (enable / disable)
        AK黑屏操作工具->>PCM_Controller服务: 更新指定AK状态 (/operation/updateAkStatus)
        PCM_Controller服务-->>AK黑屏操作工具: 返回更新结果
    else 查询账号AK (query)
        AK黑屏操作工具->>PCM_Controller服务: 查询AK列表 (/initAK/queryInitAkList)
        PCM_Controller服务-->>AK黑屏操作工具: 返回AK列表数据
        AK黑屏操作工具->>AK黑屏操作工具: 过滤并匹配指定账号ID的AK
    end
    
    AK黑屏操作工具-->>用户: 输出操作结果及统计信息
```
