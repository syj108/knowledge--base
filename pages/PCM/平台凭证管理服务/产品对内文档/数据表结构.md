# 数据表结构

- **所属服务**：`baseService-umm-ak`
- **数据库实例**：`ummak`

#### accesskey_table (AK凭证表)

用于存储 AK 的核心状态与基础信息。系统限制每个 `user_id` 下最大支持 1000 把有效 AK，达到上限后会导致派生失败。

| 字段名 | 说明 | 备注/枚举值 |
| --- | --- | --- |
| `access_id` | AK ID | 凭证唯一标识 |
| `access_key` | SK (Secret Key) | 凭证密钥 |
| `user_id` | 账号 UID | 所属用户标识 |
| `enabled_flag` | 启用状态 | `1`: 启用，`0`: 禁用 |
| `hidden_flag` | 隐藏状态 | `0`: 正常显示 |
| `deleted_flag` | 删除状态 | `1`: 已删除，`0`: 正常 |
| `modified_time` | 修改时间 | UNIX 时间戳 |

## PCM 数据库 (pcm_db)

- **所属服务**：`certificate-lifecycle-manager-server`
- **数据库实例**：`clm_db`

#### init_ak_info (Init AK信息表)

用于存储 PCM 托管的底表 AK（Init AK）记录及其在 UMM 侧的同步状态。

| 字段名 | 说明 | 备注/枚举值 |
| --- | --- | --- |
| `access_key_id` | AK ID | 关联的 AK 标识 |
| `umm_ak_status` | UMM AK 状态 | `0`: 禁用 |

#### 派生 AK 记录存储机制说明

PCM 数据库会持久化保留派生 AK 的记录。对于每个派生队列，白屏界面仅支持查询最近的 14 把派生 AK；当派生 AK 数量超过 14 把时，系统会在 UMMAK 侧（`accesskey_table`）执行删除操作（置 `deleted_flag = 1`），但 PCM 数据库（`pcm_db`）中仍会保留该派生 AK 的历史记录，可用于追溯和应急查询。
